# v1.6.0 — Fix 1 redesign (utility-based inbound eviction)

**Release type:** Design-change release. Replaces v1.5.0 Fix 1 (random eviction) wholesale. Keeps v1.5.0 Fix 2 and v1.5.2 subscriber-race fix unchanged.
**Spec status:** Draft for expert review BEFORE any implementation. Do not implement until external reviewer signs off on both the threat model and the proposed mitigation.
**Empirical verification:** **Non-negotiable.** v1.6.0 cannot ship without a successful cold-bootstrap reproduction on a fresh node against the production network. See Release Gate below.

---

## What this replaces

**v1.5.0 Fix 1 (random eviction) is removed.** Specifically:
- `PeerRegistry::decide_inbound_eviction` — deleted or rewritten
- `PeerRegistry::eligible_eviction_candidates` / `_with_min_age` — deleted
- `EvictionCandidate`, `EvictionDecision` — replaced with utility-based variants
- `EVICTION_PENDING_HEADROOM = 32` constant — kept (same purpose, accept-time overcommit for post-handshake eviction to land)
- `EVICTION_MIN_AGE_SECS = 60` constant — replaced with a smaller baseline that only gates handshake-settle, not eviction target selection

**What is NOT touched:**
- v1.5.0 Fix 2 forward-chain validation — unchanged
- v1.5.2 TipResponse handler guard — unchanged
- Consensus rules — unchanged
- Wire protocol — unchanged
- `attach_session` duplicate-identity replace logic — unchanged
- `record_ip_strike` and identity-ban machinery — unchanged

v1.4.2 / v1.5.0-1.5.2 / v1.6.0 all interoperate on the same chain.

## Why random eviction failed

Observed in production on S2-prod (live Exfer node with ~256 inbound slots full):
- **68,197 evictions logged** since v1.5.0 upgrade.
- **Every single eviction victim had age 60-65s.** No exceptions.
- **Fresh nodes cannot establish sustained connections** — S2-prod reports "max inbound peers, no eviction candidates" repeatedly, rejecting incoming handshakes.
- **68 per-second reject rate** at peak observation times.

### Root cause of the thrash

With `EVICTION_MIN_AGE_SECS = 60` and random selection over `age >= MIN_AGE`:

1. A peer is admitted to a slot.
2. 60 seconds pass — it crosses the min-age threshold and joins the eligible pool.
3. Under high connection pressure (dozens of inbound attempts per minute on a popular node), **the eligible pool is rarely deep** — it consists mostly of the few peers that just crossed the 60s threshold.
4. Random selection over a small pool of same-age peers ≈ "evict whichever peer just turned 60s old".
5. The evicted peer reconnects (normal outbound-retry backoff).
6. Reconnect arrives → slot full → evicts another just-turned-60s peer.
7. Cascade: peer set asymptotes to "everyone is 60-65s old, eviction + reconnect cycle is unbreakable".

### Why "raise MIN_AGE" doesn't fix this

The failure mode is a **cliff at MIN_AGE**. Any hard threshold where "age < X ⇒ fully protected, age >= X ⇒ equally eligible with all other 'eligible' peers" produces the same pathology when pressure is high. Raising X to 300s or 1800s delays the onset but doesn't eliminate it: on any node with enough inbound pressure to create `MAX_INBOUND_PEERS / X` new connections per second, the eligible pool will always concentrate at the threshold and the cascade will recur.

Parameter tuning on a flawed mechanism is not the answer. The mechanism itself is wrong.

### Why random-over-eligible is wrong

Random selection does not penalize the peer that caused the eviction event (the new inbound connection). In a colonization scenario, the colonizer keeps dialing in; eviction cycles cost them nothing extra while it continuously churns the rest of the pool. A mechanism that targets **colonizers** rather than **whoever happens to be oldest-young** gives colonizers the cost and leaves diverse legitimate peers stable.

## Design — utility-based eviction

**Modeled on Bitcoin Core's `AttemptToEvictConnection`.** That algorithm has been in Bitcoin Core since 2014, battle-tested against every kind of eclipse and colonization attack. We adapt it to Exfer's protocol.

### Core principle

When a new inbound peer's handshake completes and slots are full, evict **the least useful peer from the most over-represented network group**, not a random peer.

"Most over-represented" = the network group with the most peers in the current slot pool (see the **NetworkGroup key definition below**). The dominant Alibaba cluster on our network (98% of crawler-visible nodes share AS45102, many in the same IPv4 /16) would be the evict-from target in that over-represented case. A diverse new connection from a different group forces a colonizer out, not a random peer.

"Least useful" within the target group = the newest peer in that group. The intuition: an older peer in the group has already proved itself over time; a newer one is the latest arrival from the same colonizer, still unproven.

### NetworkGroup key definition

The brief requires a canonical group-key type that encodes the grouping rule deterministically for both IPv4 and IPv6:

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub enum NetworkGroup {
    /// IPv4 /16 — first 2 octets of the IPv4 address.
    Ipv4Slash16([u8; 2]),
    /// IPv6 /32 — first 4 bytes of the IPv6 address.
    Ipv6Slash32([u8; 4]),
    /// Loopback traffic gets its own group so it isn't bucketed with
    /// either public IPv4 or IPv6.
    Loopback,
}

impl NetworkGroup {
    pub fn from_ip(ip: std::net::IpAddr) -> Self {
        match ip {
            std::net::IpAddr::V4(v4) if v4.is_loopback() => NetworkGroup::Loopback,
            std::net::IpAddr::V6(v6) if v6.is_loopback() => NetworkGroup::Loopback,
            std::net::IpAddr::V4(v4) => {
                let o = v4.octets();
                NetworkGroup::Ipv4Slash16([o[0], o[1]])
            }
            std::net::IpAddr::V6(v6) => {
                let o = v6.octets();
                NetworkGroup::Ipv6Slash32([o[0], o[1], o[2], o[3]])
            }
        }
    }
}
```

The group key is the HashMap key for the group-pass of the algorithm (step 2 below) and the candidate field on `EvictionCandidate`. The ordering (`PartialOrd, Ord`) is used for deterministic tiebreaking.

### Algorithm (precise)

On the post-handshake eviction decision path, under the peers lock:

1. **Protection pass — exclude peers from eviction consideration:**
   - **Active IBD peer** (identity + session_id matches `active_ibd_peer`) — same as v1.5.0.
   - **Peers in handshake settle window** — session age < `POST_HANDSHAKE_GRACE_SECS` (proposed **15s**, down from v1.5.0's 60s). Gives handshake + initial GetTip/Addr exchange room to complete without churn, but is short enough not to gate the main selection algorithm.
   - **Top-N by recent useful-message timestamp** — peers that have sent us a recently-useful message (NewBlock, BlockResponse during IBD, TipResponse that passed forward-chain validation, Addr that contained new IPs). Protects `PROTECT_USEFUL_N = 8` most-recently-useful peers.
   - **Top-N by session age** — protects `PROTECT_OLDEST_N = 8` peers with the longest sustained session. These have proven stability.
   - **Top-N by network-group diversity** — *deterministic selection rule:* for each unique `NetworkGroup` represented in the inbound pool, identify that group's oldest-established member. Sort those groups by the session age of their oldest member (oldest first); break ties by the `NetworkGroup` key's canonical byte-wise order (`Ord` impl on the enum). Protect the top `PROTECT_GROUPS_N = 16` groups' oldest members. If fewer than 16 groups exist in the pool, protect all of them. This favors groups whose anchoring peer has been with us longest — rewarding long-tenured diversity rather than newly-appeared exotic tails. The hard cap prevents a small-subnet colonizer from hiding each of their owned /16s under diversity protection — once they exceed `PROTECT_GROUPS_N`, they start competing for protection slots against each other.
   - **Outbound peers** — never considered for inbound eviction (they're not in the inbound count).

2. **Group pass — identify the over-represented subnet:**
   - Group all non-protected inbound peers by `NetworkGroup` (IPv4 /16 / IPv6 /32 / Loopback).
   - Find the group with the **largest membership**. Tiebreak ladder, applied in order until one group wins:
     (a) highest total session age across the group's members — penalize long-entrenched colonizers
     (b) smallest `NetworkGroup` by canonical `Ord` byte-order (the `NetworkGroup` enum's derived `Ord`)
   - The single group that survives the ladder is the "target group".
   - **Fully-diverse fallback:** if the largest group has only 1 peer (every non-protected peer is in a group of its own — maximally diverse pool, no over-representation to exploit), skip the group pass entirely and proceed as if the target group is "the entire non-protected pool". Victim selection in step 3 then picks the newest peer across all non-protected peers. (No /8 fallback; a coarser bucketing would reintroduce an IPv6 mismatch and isn't needed — in a singleton-per-group state there is no colonizer to target.)

3. **Victim pass — select within target group:**
   - Within the target group, the victim is the **newest non-protected peer** (smallest session age among non-protected members of the group). Tiebreak ladder for exact-equality cases (two peers with indistinguishable `established_at` under the clock's resolution), applied in order until one peer wins:
     (a) smallest `session_id` (the monotonic counter minted at handshake; the lower session_id was established first in a strictly-ordered sense even when `Instant` resolution ties)
     (b) smallest `identity` by byte-order (`PeerId` = `[u8; 32]`, so `Ord` is lexicographic — always total)

   This makes selection fully deterministic: for any fixed peer-registry state, the same victim is selected every time the algorithm runs. Important for test stability and for reasoning about behavior under replay.

4. **Admission atomicity** — same as v1.5.0: victim selection and new-peer admission run under a single peers-lock acquisition, plus a re-check of `inbound_count() >= MAX_INBOUND_PEERS` at the top (handles the race where two handshakes complete simultaneously and only one slot is free).

   **Implementation note (concurrency tightening):** in the current code path, `active_ibd_peer` is snapshotted *outside* the peers lock (at `sync.rs:2876` and `sync.rs:5012`) and passed into the decision function. That means the IBD-protection check and the peer-pool mutation are not perfectly atomic — a peer could theoretically become the active IBD peer between the snapshot and the eviction decision. The window is very narrow in practice (IBD start happens once per sync cycle) and the worst-case outcome is that the IBD peer could be evicted mid-start. Implementation should tighten this by either (a) holding the `active_ibd_peer` lock concurrently with the peers lock during `decide_inbound_eviction_utility`, or (b) re-checking `active_ibd_peer` after acquiring the peers lock. Either approach is acceptable; the spec just requires that the eviction selection and IBD-peer-protection check are consistent within a single critical section.

5. **Frame budget release via Drop** — unchanged from v1.5.0 Fix 1 (v1.5.2 brief text already covers the invariant).

### Why this avoids the thrash

- **A diverse new peer (different /16)** joining does NOT kick the "most recently turned 60s" peer — it kicks a peer from the over-represented subnet. Diverse peers accumulate and stay.
- **A colonizer from the over-represented /16** joining kicks another peer from **their own /16** — the colonizer pays the cost internally. Their total slot count can't grow, and doesn't thrash the rest of the pool.
- **The protection slots** (oldest-N, useful-N, group-diversity-N) ensure the base population is stable regardless of pressure.
- **No age cliff**: the POST_HANDSHAKE_GRACE_SECS is a brief warmup (15s), after which peers enter the scoring pool. Scoring is continuous on session_age, not a threshold flag.

### Useful-message tracking

Each `LogicalPeer` gets a new field: `last_useful_message_at: Option<Instant>`. Updated when the peer sends:
- `NewBlock` of a block we don't already know (valid PoW, valid structure) — actual progress
- `BlockResponse` during IBD (sync peer contribution)
- `TipResponse` whose forward-chain validation reached `confirmed: true`
- `Addr` containing at least one IP not already in our address book

Initial value: `None` (peer hasn't proved useful yet). Peers with `None` are eligible for eviction; peers with `Some(recent_t)` get protection until the usefulness times out (`USEFUL_PROTECTION_SECS`, proposed **600s** — 10 min).

**Session scoping (required):** `last_useful_message_at` is **session-scoped** — it belongs to the current session, not to the persistent logical peer identity. A `LogicalPeer` survives disconnects and session replacements (see `attach_session` at `src/network/sync.rs:170`), so the field must be explicitly reset whenever a new session is attached.

**The rule is uniform: reset `last_useful_message_at = None` on every successful `attach_session` call, regardless of the `SessionAttachResult` variant, before the new session is visible to the eviction selector.** Both paths need it:

- `SessionAttachResult::NewLogicalConnect` fires whenever `lp.session.is_none()`. In the current code (`src/network/sync.rs:313`) that `or_insert_with` + `lp.session.is_none()` branch covers two cases:
  (a) a genuinely new `LogicalPeer` created by the `or_insert_with` call (first-ever connection from this identity), and
  (b) an existing `LogicalPeer` whose prior session was detached — i.e., a peer reconnecting after disconnect. The LogicalPeer object persists across disconnects and retains its fields.
  In case (b), `last_useful_message_at` from the previous session would carry over if we don't explicitly clear it. A reconnecting peer must start the new session with `None`, not inherit the old session's useful-message credit.
- `SessionAttachResult::ReplacedExistingSession` (`src/network/sync.rs:344`) — old session's usefulness is discarded. Set `last_useful_message_at = None` as part of the session swap.

Rewriting the contract compactly: **every new session starts with `last_useful_message_at = None`.** The eviction selector may only see a `Some(t)` value when the current session itself has sent a useful message. This is the invariant "the newest peer in the target group is the least proven" relies on.

// Addition to LogicalPeer (in src/network/sync.rs)
pub struct LogicalPeer {
    // ... existing fields ...

pub struct EvictionCandidate {
    pub identity: PeerId,
    pub session_id: u64,
    pub socket_addr: SocketAddr,
    pub established_at: Instant,

pub enum EvictionDecision {
    NotNeeded,
    DuplicateIdentity,
    IpCapReached,
    NoEligibleCandidates,  // every peer is protected — admit new peer impossible
    Evict(EvictionCandidate),

impl PeerRegistry {
    // Replaces v1.5.0's decide_inbound_eviction. Utility-based selection.
    pub fn decide_inbound_eviction_utility(
        &self,
        new_peer_identity: &PeerId,
        new_peer_ip: IpAddr,

        max_inbound: usize,
        config: &EvictionConfig,
    ) -> EvictionDecision { ... }
}

pub struct EvictionConfig {
    pub post_handshake_grace_secs: u64,  // default 15
    pub protect_useful_n: usize,           // default 8
    pub protect_oldest_n: usize,           // default 8
    pub protect_groups_n: usize,           // default 16

}

## New constants (src/types/mod.rs)

```rust
/// Handshake + initial-exchange settle window. Peers younger than this are
/// protected from eviction selection. Smaller than v1.5.0's EVICTION_MIN_AGE_SECS
/// because eviction is now utility-based, not age-gated; this window only ensures
/// GetTip/GetAddr exchanges complete before the peer is scored.
pub const POST_HANDSHAKE_GRACE_SECS: u64 = 15;

/// Number of inbound peers protected from eviction by virtue of most-recent
/// useful-message timestamp. Top-N most-recently-useful peers are spared.
pub const PROTECT_USEFUL_N: usize = 8;

/// Number of inbound peers protected by session age (oldest first).
/// Guarantees a core of stable long-tenured peers.
pub const PROTECT_OLDEST_N: usize = 8;

/// Number of unique network-groups whose oldest member is protected.
/// Deterministic selection rule: groups ranked by age of their oldest
/// member (oldest first), ties broken by NetworkGroup byte-order. Top-N
/// groups' oldest members are protected. If the pool has fewer than N
/// groups, all groups' oldest members are protected.
pub const PROTECT_GROUPS_N: usize = 16;

/// Useful-message freshness: a peer that sent a useful message within the last
/// N seconds is eligible for the useful-protection top-N.
pub const USEFUL_PROTECTION_SECS: u64 = 600;
```

`EVICTION_MIN_AGE_SECS` is removed.
`EVICTION_PENDING_HEADROOM` is kept unchanged (still 32).

## Alternative designs considered and rejected

### (R1) Keep random-eviction, raise MIN_AGE to 1800s

Rejected. As analyzed above: same cliff, same self-reinforcing collapse pattern under high pressure, just on a longer timescale. Patches around the symptom, doesn't fix the mechanism. User explicitly rejected this (the driving reason v1.5.3 was skipped).

### (R2) Keep random-eviction, add eviction rate limit (e.g., 1 per 30s)

Rejected. Protects against the cascade on paper but introduces a new problem: during legitimate heavy inbound activity (e.g., nightly connection storms from the Alibaba fleet), genuine new peers can't get in because every slot is full and eviction is rate-limited. The cap is an arbitrary parameter with no connection to the colonization or diversity objectives.

### (R3) No eviction — revert to v1.4.2 "reject when full"

Rejected. This gives up the core colonization-resistance property that motivated Fix 1 in the first place. The Alibaba fleet would happily fill every slot FCFS and keep them forever, recreating the exact situation v1.5.0 was meant to fix. Status quo ante is not an option.

### (R4) Deterministic "evict oldest" instead of random

Rejected. User originally pushed back against deterministic selection in the v1.5.0 design phase on fingerprinting-surface grounds (an attacker could predict eviction targets). Utility-based selection is deterministic but based on per-peer observations that the attacker can't easily game without actually being useful — a much narrower fingerprinting surface.

### (R5) Pure "longest-ping-latency wins" (like Bitcoin Core's AttemptToEvictConnection default path)

Partially adopted (the "recent useful message" protection slot is spiritually similar). We don't currently track ping latency continuously for every peer, so that specific metric isn't available. If we add latency tracking in the future, the protection slot criterion could be expanded. Not blocking for v1.6.0.

## Empirical release gate (non-negotiable)

Before v1.6.0 can be tagged, ship, or published:

1. **Local lib + integration test suite green.** (`cargo test` exit 0)
2. **Fresh-node cold-bootstrap test, end to end, against the live production network.**
   - On a clean datadir on one of our servers (or any clean Linux box).
   - Start the v1.6.0 binary pointed at ≥3 seed peers.
   - Observe: node completes forward-chain validation of the first peer, enters IBD, reaches network tip.
   - Expected completion time ≤ 6 hours on 4+ cores (Argon2-bound; per v1.5.0 cold-bootstrap analysis).
   - **Hard gate:** if the node does not reach tip within 8 hours, DO NOT SHIP. Investigate. Re-verify.
3. **Regression test on existing v1.5.2 + v1.4.2 cold-bootstrap cases** (documented in v1.5.2-brief.md and v1.5.0-brief.md regression tests): must still pass.
4. **A v1.6.0 node upgrading in place on an existing v1.5.x chain state** must resume normally (the eviction algorithm is node-local; restart just swaps the decision function).

The v1.5.0 release went out believing the spec analysis was conclusive. v1.5.2 shipped believing static analysis of the fix was conclusive. Both turned out to have real-world failure modes we missed. **Empirical verification is the release-gate of record for v1.6.0 and all subsequent releases touching network behavior.**

## Tests required

### Unit tests (in `src/network/sync.rs` or dedicated test module)

1. **`eviction_picks_victim_from_most_over_represented_group`** — populate registry with 20 peers: 15 in /16 = 192.168, 5 in /16 = 10.0. Run eviction selection. Assert victim's IP is in 192.168/16 (the over-represented group).

2. **`eviction_picks_newest_within_target_group`** — populate registry with 10 peers in same /16, session ages 10s..100s (all past POST_HANDSHAKE_GRACE). Run 10,000 selections. Assert the 10s peer is selected roughly 100% of the time (newest in target group = lowest age).

3. **`eviction_skips_post_handshake_grace_window`** — registry with peers at age 5s, 10s, 20s. Assert only the 20s peer can ever be selected.

4. **`eviction_protects_oldest_n_peers`** — registry with 20 peers, ages 10min..100min. Assert the top `PROTECT_OLDEST_N = 8` oldest are never selected (i.e., only peers with rank 9-20 by age can be victims).

5. **`eviction_protects_useful_n_peers`** — registry with 20 peers, 8 of which have recent `last_useful_message_at`. Run eviction. Assert the 8 useful-recent peers are never selected.

6. **`eviction_protects_group_diversity_representatives`** — registry with 20 peers across 5 distinct /16 subnets. Assert the oldest peer in each /16 is protected (subject to the overall cap `PROTECT_GROUPS_N`).

7. **`eviction_falls_back_gracefully_when_all_peers_protected`** — registry with only protected peers (e.g., all 20 are in oldest-top-8 category). Assert selection returns `NoEligibleCandidates` cleanly, not a panic.

8. **`eviction_respects_active_ibd_peer_protection`** — as per v1.5.0 Fix 1 test; unchanged.

9. **`eviction_under_high_churn_does_not_thrash_the_pool`** — stress test: simulate 1000 connection attempts over 60 seconds to a 256-slot pool (the same pressure rate that produced the v1.5.0 thrash). At the end, assert **only the age-based thrash-avoidance properties**:
   - Mean peer age ≥ 5 minutes (v1.5.0 broken state showed mean 60-65s; this would fail loudly on regression)
   - No more than 20% of peers are younger than 30 seconds at any sample point during the run (pool should not be dominated by just-admitted peers)
   - Eviction rate stabilizes at a bounded value (no runaway cascade — but no specific numeric cap; the pattern is what matters)

   **Colonization fairness (pool diversity under skewed-but-honest arrivals)** is NOT asserted here — under an honest arrival distribution that is itself 70% one /16, a steady-state pool reflecting that distribution is correct behavior, not a bug. Colonization-resistance is asserted specifically by tests 10 and 12, where the arrival-rate ratio is controlled.

10. **`eviction_colonization_resistance_against_single_operator_fleet`** — stress test: fleet A (1 /16) tries to fill all slots over 60s; fleet B (diverse /16s) has 1/10 of A's dial rate. Assert fleet B's slot share stabilizes at ≥ its arrival-rate share (~9%, the same property as the v1.5.0 test). Must NOT stabilize at near-zero as it would without eviction. This is the **colonization-resistance property** that v1.5.0 Fix 1 was meant to provide, carried forward.

### Integration tests (new file `tests/v1_6_0_eviction_integration.rs`)

11. **`eviction_mechanism_avoids_thrash_cascade`** — the direct regression test for v1.5.0's failure mode. Simulate the exact scenario: 256-slot pool, continuous incoming attempts at ~5/sec. Run for 5 minutes simulated. Assert:
    - Mean peer age ≥ 2 minutes (vs. v1.5.0's observed 60-65s)
    - No more than 20% of peers are younger than 30s at any sample point
    - Eviction rate stabilizes at ≤ 1/sec (vs. v1.5.0's uncapped)

12. **`colonization_resistance_long_horizon`** — port of the v1.5.0 Fix 1 long-horizon test to utility-based eviction. Same methodology (10 trials × 60 per-second samples), same pass criterion (fleet_B share within ±30% of arrival-rate share), now with utility-based selection. Must pass.

### Release-gate empirical test

13. **`v1_6_0_live_network_cold_bootstrap`** — documented procedure, not an automated test. Fresh datadir, v1.6.0 binary, seed peers = S1/S2/S3/one or two external. Run until tip reached. Capture log, confirm no thrash, no race, no spurious strikes. Pass/fail recorded in the release checklist.

## Rollout considerations

- **Wire-protocol compatible.** Eviction is node-local admission behavior; v1.5.x and v1.6.0 interoperate on the same chain.
- **Upgrade-in-place safe.** Nodes restart from existing storage with the new eviction algorithm swapped in; no state migration.
- **Rollback safe.** If v1.6.0 has some field issue, rolling back to v1.4.2 or v1.5.2 is a simple binary swap. The pre-v1.5.0 "no eviction" behavior and v1.5.2's "random with 60s min-age" behavior are both deployable fallbacks.
- **Parameter tuning is cheap post-release.** All protection thresholds (`PROTECT_USEFUL_N`, `PROTECT_OLDEST_N`, `PROTECT_GROUPS_N`, `USEFUL_PROTECTION_SECS`, `POST_HANDSHAKE_GRACE_SECS`) are tunable constants. If production data suggests a better balance, a v1.6.x patch release can tweak them without the larger structural change.
- **Expert re-review on the admission atomicity section.** The new utility-based selector still must run under a single peers-lock acquisition, and the victim-selection must not allow races with concurrent handshakes completing. Reviewer should confirm.

## Release-notes draft (for user-facing announce)

> **Exfer v1.6.0 — inbound eviction redesign**
>
> v1.5.0's random-inbound-eviction mechanism (Fix 1) was found to degrade under high connection pressure on popular nodes: every peer cycled through the pool at ~60 seconds, and fresh nodes couldn't establish sustained connections for cold bootstrap. v1.6.0 replaces Fix 1 wholesale with a utility-based selector modeled on Bitcoin Core's eviction algorithm.
>
> New behavior: when a slot needs to free up, the node evicts the newest peer from the most over-represented /16 subnet in its inbound pool, subject to protections for the longest-tenured peers, most-recently-useful peers, and per-subnet diversity representatives. This makes colonizers pay the cost of their own colonization (their peers get churned internally), leaves diverse legitimate peers stable, and eliminates the 60-second thrash cycle that was blocking fresh-node cold bootstrap in v1.5.0–v1.5.2.
>
> Fix 2 (tip-hardening) and v1.5.2's subscriber-race fix are unchanged. No consensus/protocol change.
>
> **Recommended for:**
> - New installs (first time using Exfer) — use v1.6.0, not v1.5.x.
> - Existing v1.5.x users — upgrade at your next convenient maintenance window. Restart costs ~4-20h of chain replay depending on hardware (design-wart noted; improvement in future release).
>

## Expert review routing

The expert already knows the Fix 1 threat model (colonization resistance) and the Fix 2 architecture. Key things to review:

1. **Is the utility-scoring model appropriate for Exfer's protocol?** The Bitcoin Core algorithm assumes certain kinds of per-peer observability (ping latency, tx/block recency, network-group mapping). Exfer's protocol is smaller; we've trimmed the scoring to signals we actually can observe. Confirm this subset is sufficient.

2. **Is `POST_HANDSHAKE_GRACE_SECS = 15` sufficient?** Handshake takes up to 5s (`HANDSHAKE_TIMEOUT_SECS`); initial GetTip + GetAddr exchange + TipResponse arrival is another ~1-2s typically. 15s leaves room but isn't 60s. Confirm.

3. **Is the "newest within over-represented group" target-within-target the right choice?** Alternative: oldest within over-represented group (punishes entrenched colonizers). But newest-in-group matches Bitcoin Core and has the property that a single attacker can't keep pushing the same IP back into the slot after it's evicted (they'd have to rotate IPs, which /16-diversity penalizes). Reviewer confirm or propose.

4. **Are the protection-slot counts (8/8/16) appropriate?** Together they reserve ~32 of 256 slots from eviction consideration. That's 12.5% of slots guaranteed to be stable-regardless-of-pressure. Reviewer confirm or propose adjustment.

5. **Any failure modes the spec misses?** The random-eviction failure mode was not obvious from spec review — it emerged from production. Reviewer should flag any comparable concerns about the new algorithm.

6. **Empirical release-gate procedure.** Confirm the cold-bootstrap test is the right gate and the 8-hour deadline is reasonable. If the reviewer believes the test should be run against specific seed peers or with a longer observation window, adjust before sign-off.
