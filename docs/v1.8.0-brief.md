# v1.8.0 — Cold-bootstrap redesign: bulk-headers anchor authentication, skip Argon2 pre-anchor only after SHA-linkage is verified

**Release type:** Design-change release. Replaces Fix 2's fragile pre-IBD gauntlet with a **bulk-header fetch followed by SHA-linkage verification backward from the hardcoded `ASSUME_VALID_HASH` anchor.** Only after the SHA-linkage is fully verified do we skip Argon2 for pre-anchor blocks — which means an attacker cannot DoS us by synthesizing a linked-but-forged chain (round-3's bug). Bundles v1.7.2's signed-off permissive-transport policy. Mirrors Fix 2's `authenticate_prechckpt_headers` security property but replaces the multi-batch gauntlet with a single bulk operation.

**Spec status:** Round 10 — revised after round-9 review flagged a parallel blocker in Stage B: the same peer-side response-byte budget that drove round-9's Stage A pacing also applies to GetBlocks, so Stage B's throughput estimate was unsupported. Round-10 extends round-9's pacing pattern to Stage B and bumps the gate accordingly. Details below.

**Round-10 fixes (condensed, scroll down for the in-body changes):**

A peer that hits its budget mid-chunk just stops, and we hit our 120-s per-block deadline without a clean "peer-paced" signal.

## What this fixes

v1.7.2 is signed-off and eliminates strike accumulation during pre-confirmation bootstrap (zero strikes in the 10-min gate run). But the peer pool it preserves can't complete an end-to-end Fix 2 validation; 31 × `NoSlotAvailable`, 11 × `peer disconnected mid-validation`, 1 × `fetch timeout`, 0 successes, height stays at 0.

v1.4.2 didn't have the Fix 2 gauntlet. It did full Argon2 IBD from any peer that advertised a tip. Took hours but worked. That was the baseline.

**v1.8.0's move:** do header authentication via a single bulk-fetch + SHA-linkage-to-anchor operation, not a multi-batch gauntlet. Once headers are authenticated, skip Argon2 for pre-anchor blocks safely. Post-anchor IBD is unchanged.

## Background — why round-3's skip-Argon2 was wrong

Argon2 PoW verification in Exfer is what prevents an attacker from cheaply fabricating a chain. Without Argon2 checks, producing a syntactically-valid block just requires:
- Pick a `prev_block_id` (any 32 bytes, but for linkage it must be the hash of the previous header in the fake chain — computable cheaply)
- Pick a `difficulty_target` (any value; no one checks if the block's hash actually meets it, since we skipped PoW)
- Build a syntactically-valid tx structure (coinbase-only for early blocks — trivial)
- Compute `block_id = SHA256(header)` — cheap

An attacker serving such a chain to a v1.8.0-round-3 victim:
- Victim IBDs optimistically, skips Argon2 on every block
- Victim does full tx-merkle / UTXO / SMT / storage work on 302K fake blocks (~10 min of serial CPU work, ~500 MB of disk I/O, ~300 MB of network)
- Victim reaches `ASSUME_VALID_HEIGHT`, compares stored block_id to `ASSUME_VALID_HASH`, mismatch
- Victim wipes datadir, strikes attacker (one strike), tries next peer
- Attacker reconnects from a different IP/identity, repeats

Asymmetric: attacker spends cheap SHA work + bandwidth, victim spends minutes of CPU + GBs of disk I/O per cycle. Pool of colluding attackers can keep a fresh node in an infinite wipe-retry loop.

Bitcoin Core doesn't have this lane because its `assumevalid` skips **script validation only**, not PoW. Per-block PoW (SHA-256d, cheap) is always verified. An attacker fabricating a Bitcoin chain has to actually find PoW, which is economically infeasible at mainnet difficulty.

For Exfer with Argon2id, per-block PoW verification is expensive (~12 ms single-core), so we can't verify every pre-anchor block cheaply. The actual safe optimization is to skip per-block PoW **after** establishing the chain's authenticity via a different mechanism — specifically, the SHA-linkage-back-to-anchor walk that Fix 2's `authenticate_prechckpt_headers` already uses.

## Design

Two changes. Change 1 is the primary redesign. Change 2 re-bundles v1.7.2's signed-off policy.

### Change 1 (primary): Bulk-header anchor authentication, then IBD blocks with skip-Argon2

**Uses the existing `ASSUME_VALID_HEIGHT` / `ASSUME_VALID_HASH` from v1.7.0. No new constants.**

Bootstrap flow for a fresh node:

1. Fresh node resolves `seed.exfer.org`, connects to peers. Peer P sends `TipResponse(height=H, block_id=B, cumulative_work=W)`.
2. **Peer classification** (no trust in peer-supplied `W`):
   - `H < ASSUME_VALID_HEIGHT`: P cannot serve us a chain that reaches the anchor. Skip as IBD source. Keep connection for relay. No strike.
   - `H >= ASSUME_VALID_HEIGHT`: P is a candidate. `tip.confirmed = false`. No `ever_confirmed_peer` flip yet. Peer enters the pre-anchor scheduler (below).
3. **Pre-anchor scheduler** (unchanged from round-3): round-robin across candidates, in-memory exponential backoff on disconnect/timeout (30s → 32m cap), strike only on the `AnchorMismatch` outcome defined below. No ranking by claimed W.
4. **Stage A — bulk-header fetch and SHA-linkage authentication** (new, replaces the Fix 2 per-batch gauntlet):

   **Transport (round-9 revised — paced single-peer, no protocol change):**
   - Sync manager picks one candidate peer P.
   - Pre-compute the request schedule: **inclusive range [0, ASSUME_VALID_HEIGHT]** — `N = ceil((ASSUME_VALID_HEIGHT + 1) / MAX_GETBLOCKS_ITEMS)` requests. For `ASSUME_VALID_HEIGHT = 302,400` (a multiple of 64) and `MAX_GETBLOCKS_ITEMS = 64`: `N = 4,726`, where request `4725` asks for `start_height = 302400, max_count = 64` and the peer returns at least `header[302400]` (the anchor). Request `i` asks for `start_height = i * MAX_GETBLOCKS_ITEMS`, `max_count = MAX_GETBLOCKS_ITEMS`. A previous version of this spec said `N = 4725` and missed the anchor — round-6 correction.
   - **Paced send with small outstanding-request window** (round-9, replaces round-5's "pipeline all N back-to-back"):
     - Maintain a bounded window of **up to 8 in-flight** `GetHeaders` requests (for RTT hiding; 8 is small enough not to stress the peer's receive side, large enough to amortize RTT across a session). A slot opens when a response is received and validated; the next request goes out.
     - **Response-byte pacing ceiling: 14 MiB/min per peer** (conservative margin below the peer's `MAX_RESPONSE_BYTES_PER_MIN = 16 MiB` cap at `sync.rs:3947`). The peer measures **serialized** message bytes (not raw header payload) at `sync.rs:3944`, so "header payload only" math is too optimistic — 14 MiB/min leaves headroom for HMAC/length-prefix/Vec-encoding framing overhead.
     - Implementation: maintain a sliding-window rate estimate of **expected serialized response bytes** per peer over the last 60 s. Before opening a new in-flight slot, check that projecting the next response onto the window stays under 14 MiB/min; if not, delay the send until the projected rate drops under the ceiling. Conservative estimate for serialized response size: ~200 B per header × 64 headers + ~100 B framing ≈ 12.9 KB per Headers message. Max rate at that size: 14 × 1024² / 12,900 ≈ 1,139 Headers/min = ~19/s. The pacing will converge to roughly this rate.
     - TCP in-order delivery at the application level is what we rely on; peer processes each `GetHeaders` in arrival order. The 8-window is just RTT hiding, not a protocol change.
   - **Receive Headers responses and correlate by `header[0].height`** (the first header's height field tells us which request this response was for). Every `Message::Headers(Vec<BlockHeader>)` is self-describing via the `height` field on each BlockHeader — no protocol change needed to add request IDs. On correlation, the matching in-flight slot is released and the next paced send is scheduled.
   - **Empty `Headers(Vec::new())` response handling** (round-9 tightens round-7 wording): an empty response can be produced by (a) the peer exceeding its response-byte budget, (b) the peer lacking the requested height range, (c) correlation ambiguity from protocol-level limitations. From the client's point of view, none of these is a provable peer lie. **Outcome is always: abort Stage A + scheduler backoff against this peer + try another peer, no strike.** The paced send strategy is designed to keep us well under the budget, but if we still see empty responses, we don't try to diagnose the cause — we just back off and retry on another peer. Strikable outcomes in Stage A remain only `delivered-invalid-header` (SHA-linkage mismatch) and `delivered-forged-chain` (anchor mismatch at `ASSUME_VALID_HEIGHT`).
   - **Subscriber:** `tip_validation.rs`'s existing subscriber plumbing (`install_headers_subscriber`, `route_headers_if_subscribed`, `await_header_batch` at `tip_validation.rs:581`, `:605`, `:625`) already forwards one `Vec<BlockHeader>` per `Message::Headers` and yields one Vec per `recv` call. No refactor needed — Stage A's per-response correlation logic plugs in on top of this as-is. (Round-5 claimed a flattening refactor was required; that claim was incorrect and is removed in round-8.)

   **Validation as headers arrive** (streamed, no buffer-all-then-verify):
   - For each received batch B with `B[0].height == expected_start_height` for some pending request:
     - For i in 0..B.len()-1: `B[i+1].prev_block_id == B[i].block_id()`. Mismatch → strike peer (delivered-invalid-header), abort Stage A.
     - Commit B to the in-memory `headers: Vec<BlockHeader>` at the corresponding position.
   - At the seam between two batches B_prev and B_next (adjacent in height): `B_next[0].prev_block_id == B_prev[B_prev.len()-1].block_id()`. Mismatch → strike, abort.

   **Anchor check at end of Stage A:**
   - After all N responses received and seam-checks pass: `headers[ASSUME_VALID_HEIGHT].block_id() == ASSUME_VALID_HASH`. Mismatch → strike peer (delivered-forged-chain class), abort, scheduler picks another peer. Match → proceed to Stage B.

   **What Stage A does NOT do:** it does NOT run Argon2 per header. The authentication comes from the SHA-linkage walk + the hardcoded-hash match at the anchor, not from per-header PoW. **Safe because:** producing a chain whose final header at height `ASSUME_VALID_HEIGHT` hashes to exactly `ASSUME_VALID_HASH` would require finding SHA-256 second-preimages for each intermediate height — cryptographically infeasible.

   **Cost estimate (round-9 revised — pacing-bound, not network-bound):** 302K headers × ~200 B serialized per header + per-message framing ≈ ~60 MB of serialized response bytes. Under 14 MiB/min peer-side pacing ceiling: **~3–4 min wall-clock Stage A** on any healthy peer. Residential network bandwidth is not the bottleneck (peer caps at ~234 KB/s well below residential downlink); Stage A throughput is bound by the peer's per-minute response-byte budget, which resets on a 60-s tumbling window at `sync.rs:3641`. Client-side parse + SHA-linkage check: ~150 ms total (negligible). Round-8 estimated ~5–10 s by ignoring the budget; round-9 corrects to ~3–4 min.

   **Timeout (round-9 revised):** **300 s (5 min) overall Stage-A wall-clock**, replacing round-8's inherited `TIP_VALIDATION_BATCH_TIMEOUT_SECS = 120 s`. Reasoning: Stage A's expected duration is ~3–4 min under pacing, so 120 s is too tight; 300 s gives ~25–40% margin over the design estimate and still acts as a strong "this peer is not viable" cutoff. Either introduce a new constant `STAGE_A_TIMEOUT_SECS = 300` in `src/types/mod.rs`, or hardcode in the coordinator — implementation detail. On timeout: abort Stage A, scheduler marks peer for backoff, picks another. No strike.
5. **Stage A → Stage B handoff — storage and flag semantics** (round-7 rewrite of round-6 step 5, adds the missing store-vector-on-Node requirement that closes round-6's P0):

   Three state updates fire at Stage A anchor-check success, one at Stage B anchor-apply. The storage update (5a) is new in round-7 and is the load-bearing one for the P0 fix.

   - **5a. Store the authenticated header vector on the Node (round-7, new).** Add a new Node field `stage_a_authenticated_headers: tokio::sync::RwLock<Option<Arc<Vec<BlockHeader>>>>` (or equivalent — the Arc/RwLock choice is implementation detail; requirement is "shared-read access, written exactly once per process lifetime at this point"). At Stage A anchor-check success, write the full Stage-A-verified `Vec<BlockHeader>` (indexed by `height`, covering `0..=ASSUME_VALID_HEIGHT`) into this field under the write lock. This vector is then the **authoritative source of `expected_id`** for Stage B below the anchor. Ordering: this write MUST complete before `assume_valid_verified` is flipped (step 5b), because `run_ibd`'s below-anchor path (step 6) reads the vector on entry and the flag gates `ibd_skip_pow`. Use `SeqCst` on the flag and a `Release` on the vector's write lock; the read path in `run_ibd` takes the `Acquire` under the read lock before any `Ordering::SeqCst` load of `assume_valid_verified`. (Or more simply: do the vector write first, then the flag flip — both are single-writer in a single `finalize_stage_a` function.)
   - **5b. At Stage A anchor-check success, flip `assume_valid_verified.store(true, SeqCst)`.** This is the gate for the existing IBD `ibd_skip_pow` check at `sync.rs:4906` (`node.assume_valid && node.assume_valid_verified.load(SeqCst) && block.header.height <= ASSUME_VALID_HEIGHT`). Flipping it here is what lets Stage B's pre-anchor block download skip Argon2. **Semantic expansion:** in v1.7.x this flag meant "we have observed the anchor block in our stored chain." In v1.8.0 it means "the anchor is cryptographically authenticated, either via Stage A's header-chain SHA proof OR via organic IBD reaching `ASSUME_VALID_HEIGHT`." The new path is strictly stronger — Stage A's SHA-linkage-to-hardcoded-anchor walk provides the same assurance that organic-IBD-reach would (since both culminate in a block whose block_id equals the hardcoded `ASSUME_VALID_HASH`). Existing downstream consumers of `assume_valid_verified` (mainly the `ibd_skip_pow` check) are satisfied, *provided the P0 rewire of step 6 is in place so Stage B's `expected_id` comes from the Stage A vector*.
   - **5c. `ever_confirmed_peer` stays false at Stage A success.** It flips only at Stage B's anchor-apply point (step 7). Reason: `ever_confirmed_peer` gates the v1.7.2 permissive-transport policy and is paired with peer-session reputation. A Stage A cryptographic proof does not yet prove that any specific peer has successfully delivered chain state; that's what the anchor-apply-during-Stage-B moment establishes. Keeping `ever_confirmed_peer` false through all of Stage B preserves v1.7.2's protection over Stage B's long block-download phase (no SlowPeer / RateLimitExceeded strikes on residential flake during IBD).

6. **Stage B — rewire `run_ibd` below the anchor to source `expected_id` from the Stage A vector** (round-7, closes round-6 P0):

   **The problem being closed.** Current `run_ibd` at `sync.rs:4787` issues `GetHeaders(start_height, max_count=64)` per batch, validates SHA-linkage locally within the batch plus the seam to `prev_batch_tip`, and derives `expected_id` per block from those *freshly-fetched-from-the-peer* headers at `sync.rs:4851`. Below `ASSUME_VALID_HEIGHT`, `process_block_pre_validated` at `sync.rs:2213` skips Argon2 solely because `assume_valid_verified = true`. Nothing in this path compares the delivered block against the Stage-A-authenticated header vector. A malicious Stage-B peer could serve a forged chain whose block_ids match its own just-fetched (also-forged) headers, and the skip-PoW path would process them.

   **Rewire.** In `run_ibd`, before the `while current_height <= their_height` loop, acquire an `Arc` to the Stage A header vector under the RwLock read guard:
   - If `current_height <= ASSUME_VALID_HEIGHT` and `stage_a_authenticated_headers` is `Some(arc)`, enter the **below-or-at-anchor path (new)** — `<=` is deliberate: the anchor block itself must be delivered through the Stage-A-bound path, since it is the one block the whole anchor proof pins. Round-7 wrote `<` here; round-8 corrects to `<=`.
   - Else, enter the **above-anchor path (existing, unchanged from today's code)**.

   **Below-or-at-anchor path (new, round-7, entry condition corrected in round-8, pacing added in round-10):**
   - **Skip the per-batch `GetHeaders` send/recv/linkage block** at `sync.rs:4787–4848`. We already have the authenticated headers for every height in this range (`0..=ASSUME_VALID_HEIGHT`).
   - For each batch (still in `MAX_GETBLOCKS_RESPONSE = 8`-sized chunks), build `block_ids: Vec<Hash256>` by reading `stage_a_authenticated_headers[h].block_id()` for `h in current_height..=std::cmp::min(current_height + chunk_size - 1, ASSUME_VALID_HEIGHT)`.
   - **Client-side pacing (round-10, mirrors Stage A pacing):** maintain a rolling 60-s estimate of **expected serialized BlockResponse bytes** per peer (estimated via `MAX_GETBLOCKS_RESPONSE × ~1 KB block average + framing`, refined with observed actuals as blocks arrive). Before sending the next `GetBlocks` chunk, check that projecting the next chunk's expected response bytes onto the window keeps the rate under **14 MiB/min** (below the peer's 16 MiB/min cap at `sync.rs:3897`, with framing headroom). If not, delay the send until the projection drops under the ceiling. Keep up to **8 GetBlocks chunks in flight** (= up to 64 blocks in flight) for RTT hiding.
   - Send `GetBlocks(block_ids)` and for each `expected_id` in order, call the round-8-updated `recv_ibd_block` variant (see "recv_ibd_block mismatch enforcement" below). The block-id mismatch-as-hard-error is what cryptographically binds Stage B's delivered blocks to Stage A's authenticated chain: the peer can only succeed by delivering blocks matching the Stage A vector; any deviation is struck.
   - **Mid-chunk peer-silence handling (round-10):** if the serving peer's `GetBlocks` handler hits its own budget mid-chunk, the peer `break`s out of its service loop at `sync.rs:3900` or `sync.rs:3903` without sending the remaining BlockResponses. The client's `recv_ibd_block` will then hit its per-block 120-s deadline for the first undelivered expected_id. Outcome: **abort Stage B against this peer, scheduler backoff the peer, pick another peer, resume Stage B from the failed height using the Stage A vector**. **No strike.** Stage A vector stays populated; `assume_valid_verified` stays true.
   - When `current_height > ASSUME_VALID_HEIGHT` (i.e., we have just processed the anchor block at `ASSUME_VALID_HEIGHT`), switch to the above-anchor path for the remainder of the IBD.

   **`recv_ibd_block` mismatch enforcement (round-8, P1b new requirement):**

   Current `recv_ibd_block` at `sync.rs:4441` has a loop that, on a `BlockResponse` from the sync identity + session whose `block.header.block_id() != *expected_id`, falls through at `sync.rs:4474` to `process_block_event` and continues the loop. This means a malicious or confused sync peer serving a *different* block (e.g., a later-height block, or an unrelated fork block) is effectively silently ignored by Stage B — the loop just waits for the correct block until timeout. That is incompatible with the round-7 safety claim ("first mismatch rejects the peer"), so round-8 requires a behavior change for the below-or-at-anchor path:

   - **Scope.** The new behavior applies when `run_ibd` is in the below-or-at-anchor path described above (either implement it as a new `recv_ibd_block_stage_b` helper, or gate the behavior with a `strict: bool` flag on the existing function — implementation detail).
   - **New behavior.** On a `BlockResponse` event from `(sync_identity, sync_session_id)` whose `block.header.block_id() != *expected_id`:
     1. Return an error from the helper (e.g., `Err("sync peer delivered wrong block for requested expected_id")`).
     2. `run_ibd` catches the error, records a strike against the sync peer with an outcome class of `delivered-wrong-block-for-requested-id` (new variant; add it to the existing strike enum), and aborts Stage B.
     3. The Stage A authenticated header vector stays on the Node (it's still valid; the next peer from the scheduler can resume Stage B against the same vector).
     4. `assume_valid_verified` stays true; no reset.
   - **Unchanged arms.** `BlockResponse` from the *wrong* identity or a *stale* session (the `sync.rs:4476–4485` arm today) is still routed to `process_block_event` as a background event. That branch is not about answers to our in-flight `GetBlocks`; it's incidental block traffic arriving on the event channel. Leaving it untouched preserves v1.7.2 permissive-transport semantics on those paths.
   - **Already-have-this-block short-circuit** at `sync.rs:4445–4449` stays as-is — on resume, if we already stored the block at `expected_id`, the helper returns `Ok(block)` without waiting. That short-circuit is still Stage-A-bound because `expected_id` came from the Stage A vector.

   **Above-anchor path (existing):** leave `sync.rs:4787–4912` as-is. Stage A's vector doesn't cover heights above `ASSUME_VALID_HEIGHT`, so we must fetch headers from the peer; `assume_valid_verified` doesn't unlock skip-PoW for these heights (the `<= ASSUME_VALID_HEIGHT` predicate in `ibd_skip_pow` sees to that), so they get full Argon2 as today.

   **Skip-PoW safety invariant (explicit):** `process_block_pre_validated` is only reached for blocks whose `expected_id` was sourced from `stage_a_authenticated_headers`. Concretely: `ibd_skip_pow = node.assume_valid && node.assume_valid_verified.load(SeqCst) && block.header.height <= ASSUME_VALID_HEIGHT` — and by construction, `run_ibd` only sends a `GetBlocks(expected_id)` for `height <= ASSUME_VALID_HEIGHT` when that `expected_id` came from the Stage A vector. Retries on other peers preserve this invariant: the Stage A vector is process-lifetime-immutable; Stage B on peer Q uses the same vector, same `expected_id`s, same block-id check.

   **Why the global `assume_valid_verified` flag is safe with this rewire (closes round-6 P1):** the safety-critical binding is the `expected_id` source, not the flag itself. Even if Stage B fails halfway and retries on a different peer, the new peer can only satisfy the round-8 block-id mismatch-as-hard-error check by serving blocks that match the Stage-A-authenticated vector. A peer serving a different (forged) chain fails at the first block-id mismatch, gets struck (`delivered-wrong-block-for-requested-id`), and Stage B re-runs on the next scheduled peer with the same Stage A vector. No peer/session-scoping of the flag needed; no reset-on-failure rule.

   **Cost estimate (round-10 revised — paced, peer-budget bound, not CPU bound):** 302K blocks × ~1 KB each = ~300 MB = ~286 MiB download. Under 14 MiB/min client-side pacing (below the peer's 16 MiB/min cap): **~20–22 min wall-clock Stage B download**. Per-block CPU work (tx validation + UTXO + SMT + storage) is ~1–3 ms on a modern laptop, so processing fully overlaps with paced download and is not the bottleneck. The Stage A vector read is a cheap in-memory `Arc<Vec>` index lookup per block — sub-microsecond. Round-6 and round-9 estimated ~5–15 min CPU-bound Stage B; that ignored the peer-side response-byte budget. See the throughput-math section for the updated full breakdown.

7. **Post-anchor: normal IBD with full Argon2.** For blocks above `ASSUME_VALID_HEIGHT`, the above-anchor path runs — existing `run_ibd` code, unchanged. Argon2 verification runs per-block as today. Release-fresh span is ~15K blocks at ~60 H/s serial = ~4 min.

8. **`ever_confirmed_peer` flip at Stage B anchor-apply:** at the moment **Stage B applies the block at `ASSUME_VALID_HEIGHT` to storage** — i.e., when our stored chain actually contains the anchor block — `self.ever_confirmed_peer.store(true, Relaxed)` AND `lp.tip.confirmed = true` fire together. `assume_valid_verified` was already true from step 5b; no redundant flip. `stage_a_authenticated_headers` stays populated (it's a single-process-lifetime structure; releasing it is a future optimization if memory matters, not a v1.8.0 concern). This keeps `ever_confirmed_peer` paired with the existing invariant ("a peer has delivered chain state past the anchor") and Stage B's IBD phase remains under v1.7.2's pre-confirmation permissive-transport policy until this point.

**Anchor-mismatch outcome class (used in Stage A).** When a peer's delivered header chain hashes to something other than `ASSUME_VALID_HASH` at `ASSUME_VALID_HEIGHT`, this is a provable lie — no natural reorg can displace a 5K-deep anchor, and SHA-256 second-preimage is infeasible. Strike the peer's identity and IP, abort Stage A, scheduler excludes the peer. **No datadir wipe needed** — we never started IBD; no block state was written.

**The "optimistic IBD before authenticating headers" idea from round-3 is removed entirely.** We always authenticate headers first (Stage A), then run IBD (Stage B). This closes the forged-history DoS.

### Change 2 (bundled from v1.7.2): Pre-confirmation permissive transport

Signed-off v1.7.2 changes ship in v1.8.0 as-is. Protects:
- The pre-anchor scheduler's transport-level disconnect failures (no strike, just scheduler backoff)
- Stage A / Stage B's transient network flakes
- Any future path where a fresh node hits transport soft-failures before `ever_confirmed_peer` flips

No round-2+ changes to v1.7.2. See `docs/v1.7.2-brief.md` for the full spec.

### What is NOT touched

- Fix 2 forward-chain validator (`tip_validation.rs`) — unchanged. No longer the cold-bootstrap path, but retained for steady-state tip verification after the node is running and needs to validate a new peer's tip claim.
- `ASSUME_VALID_HEIGHT` / `ASSUME_VALID_HASH` / `ASSUME_VALID_CUMULATIVE_WORK` — unchanged from v1.7.0.
- Fix 1 utility eviction (v1.6.0) — unchanged.
- `ever_confirmed_peer` sticky flag (v1.7.1) — unchanged in semantics. The flag continues to mean "a peer has delivered chain state past the anchor." v1.8.0 just adds a new site where that transition happens (Stage B anchor-apply), in addition to the existing organic-IBD-reach path.
- `assume_valid_verified` atomic (v1.7.0) — **semantic widening only.** Previously flipped exclusively when IBD organically reached `ASSUME_VALID_HEIGHT`. v1.8.0 adds a second trigger: Stage A's successful header-chain SHA authentication against the hardcoded anchor. Both paths flip the same flag; downstream consumers (primarily the `ibd_skip_pow` check at `sync.rs:4906`) see the flag's meaning as "the anchor is cryptographically authenticated," which both paths satisfy. No new flag introduced.
- Consensus rules, wire protocol — unchanged.
- Above-anchor path in `run_ibd` (`sync.rs:4787–4912` for heights `> ASSUME_VALID_HEIGHT`) — unchanged. Only the below-or-at-anchor path is rewired to source `expected_id` from the Stage A vector.
- `process_block_pre_validated` / `process_block_inner` signatures and behavior — unchanged. The safety invariant is enforced by the **caller** (`run_ibd`) through the `expected_id` source, not by changes to the process-block path.
- `recv_ibd_block`'s behavior on `BlockResponse` from a *different* identity or stale session (`sync.rs:4476–4485` arm) — unchanged. It still routes to `process_block_event` as a background event.

**New additions / changes (round-7 + round-8):**
- Node field `stage_a_authenticated_headers: tokio::sync::RwLock<Option<Arc<Vec<BlockHeader>>>>` (name/type negotiable at implementation time). Written exactly once per process lifetime at Stage A anchor-check success; read by `run_ibd`'s below-or-at-anchor path.
- `recv_ibd_block`'s behavior on the **sync identity+session** arm with mismatched `block_id` is changed (round-8): from "fall through to `process_block_event` and keep looping" to "return `Err` immediately" — implemented either as a new strict-mode helper (e.g., `recv_ibd_block_stage_b`) or as a `strict: bool` parameter on the existing function. `run_ibd` records a `delivered-wrong-block-for-requested-id` strike on this error and aborts Stage B. This is the load-bearing change for the round-7 binding story.
- Strike enum gains a new variant / outcome class `delivered-wrong-block-for-requested-id`.

## Fallback: what happens if no peer has `H >= ASSUME_VALID_HEIGHT`

Explicitly decided (round-4): **the node waits.** No Fix 2 slow path. No heroics.

Why this is the right answer:
- All peers being below the anchor means `seed.exfer.org` is returning an entirely stale pool. That's a crawler / DNS operational issue, not something the node should try to route around.
- DNS is refreshed every TTL (seconds–minutes). A retry a minute later will likely produce different peers.
- The crawler on S2 produces a fresh `healthy_nodes.txt` every 10 min; new installs hit that eventually.
- Any attempt to "wait it out" via a Fix-2-style slow path just recreates the gauntlet fragility we're replacing.

Behaviour: log `"no peer is at or past ASSUME_VALID_HEIGHT yet — waiting for DNS refresh"` at INFO level once per minute. Re-resolve DNS on an exponential backoff (1 min, 2 min, 5 min, 10 min, 10 min, ...). Accept that this situation is rare and, when it happens, is usually a crawler issue that resolves on its own.

## Throughput math (release-fresh binary, ≤ 7 days old)

- **Stage A (paced bulk-header fetch + SHA-linkage + anchor check):** **~3–4 min**. Bound by the peer's per-minute response-byte budget (`MAX_RESPONSE_BYTES_PER_MIN = 16 MiB` at `sync.rs:3945–3950`), not by residential network bandwidth. Client paces at a conservative 14 MiB/min to leave framing-overhead headroom.
- **Stage B pre-anchor (paced full block IBD, Argon2 skipped, tx/UTXO/SMT runs):** **~20–22 min** (round-10 revised; round-9 said ~5–15 min but ignored that the same 16 MiB/min budget applies to `GetBlocks` at `sync.rs:3888–3919`). Same 14 MiB/min conservative client-side pacing. Payload: ~302K blocks × ~1 KB avg = ~286 MiB → ~20.4 min floor. Per-block CPU processing (~1–3 ms) runs concurrently with the next paced download, so CPU is not the bottleneck; network pacing is. v1.4.2's "hours-to-never" experience on this same budget was compounded by peer churn and the old non-paced chunk behavior hitting the peer's silent-break; round-10's client-side pacing avoids that failure mode.
- **Post-anchor IBD (full Argon2, no response-budget concern at the usual release-cut span):** ~4 min for ~15K blocks at ~60 H/s serial. Post-anchor payload ~15 MiB is under the 16 MiB/min cap — a single peer serves it within one budget window.
- **Total: ~27–30 min on a modern laptop.** Release gate is **≤ 35 min** (round-10 bump from round-9's ≤ 25 min, upper bound of the design's own estimate plus ~5-min margin). Still gate-matches-estimate per the round-5 P1c discipline; still a large UX improvement over "hours-to-never."

**Stale-binary scaling (informational, not gated):** post-anchor span grows linearly with binary age. 7-day stale = ~25 min. 30-day stale = ~95 min. Documented honestly; future release can add a bundled-snapshot shortcut.
