# v1.4.2 — Security Hardening

**Release type:** Security patch. Ship ASAP.
**Scope:** Three contained fixes from expert audit (Apr 16). No protocol changes. No new features.
**Upgrade urgency:** Wallet users should upgrade before next spend against any non-local RPC.

---

## Fix 1 — Wallet must not trust RPC-reported values (HIGH)

### The attack
Wallet calls `get_address_utxos` and `get_transaction` against an RPC endpoint, reads `value` and `script` fields from the JSON response, uses them to select inputs and compute change. A malicious or compromised RPC returns understated values. The signed transaction is valid against the honest chain, and the delta becomes miner fee. Funds are silently burned.

Three sub-issues, all in the same trust-boundary failure:

**1a.** `src/main.rs:949, 2523, 2564, 1232, 1382` and `src/wallet/wallet.rs:231` — spend paths build a local `UtxoSet` from RPC response fields directly.

**1b.** `fetch_lock_tx_output` accepts `tx_hex` from RPC without verifying `hash(tx_hex) == requested_txid`.

**1c.** Direct `HtlcClaim` / `HtlcReclaim` flows skip locked-script verification.

### Required fix

**Centralize the authenticated lookup.** All spend paths — normal wallet spend, `HtlcClaim`, `HtlcReclaim`, covenant helpers — must go through a single helper:

```
authenticated_output_lookup(
    txid: TxId,
    index: u32,
    expected_script: Option<&Script>,
) -> Result<(Value, Script), AuthError>
```

The helper:
1. Fetches the full funding transaction hex via `get_transaction`.
2. Deserializes it locally.
3. Strict parse check: verify **both** `deserialized_tx.tx_id() == txid` **and** `consumed == raw.len()`. The trailing-bytes check rejects valid-prefix-plus-garbage attacks.
4. Extracts the output at `index`. If `index` is out of bounds, abort.
5. If `expected_script` is provided, verifies exact byte-equality against the output's script. If it doesn't match, abort.
6. Returns authenticated `(value, script)`.

Any mismatch aborts with a clear, specific error (see Error semantics below).

**The JSON `value` and `script` fields from `get_address_utxos` are used only for outpoint discovery.** Treat `get_address_utxos` as returning a list of outpoints. Do not pass the JSON `value` or `script` into any spendable struct. Refactor so those fields are not reachable in the spend path — delete, don't just stop using.

**Script match requirement:** For the normal wallet path, `expected_script` is the locally-derived wallet script (exact byte equality, not "address-ish"). For covenant helpers, `expected_script` is the locally-reconstructed covenant script. For `HtlcClaim` / `HtlcReclaim`, `expected_script` is the locally-reconstructed HTLC locked script built from the claim/reclaim parameters — **and this reconstruction must use the CLI-provided timeout value**, not any default or cached value. The current code path ignores the CLI timeout; fix it.

This makes value-spoofing cryptographically impossible. The tx serialization is self-authenticating against the txid; once the strict-parse check passes, the values come from data the RPC cannot forge without breaking the hash.

### Residual trust after this fix — document in CHANGELOG and helper doc comment

With this fix, a malicious RPC can still:
- Omit UTXOs (availability attack — wallet shows lower balance)
- Return already-spent outpoints (wallet discovers at broadcast time)
- Lie about confirmation depth and tip height (UX / unsafe-spend attack)
- Lie about coinbase maturity unless height is authenticated separately

These are availability / UX issues, not theft. That's the boundary for this release. Future work (tracked, not scheduled): SPV-style inclusion proofs against a locally-maintained header chain, which would close the remaining lies at the cost of requiring the wallet to maintain headers.

### Error semantics
When RPC returns data that fails authentication, the wallet must:
- Abort the current operation
- Print a clear error distinguishing this from a network error: "RPC returned transaction data that does not match requested txid — endpoint may be malicious or compromised"
- Not retry automatically against the same endpoint

### Tests required
- Unit test: mock RPC returns `tx_hex` whose hash doesn't match requested txid → wallet aborts with the specific error
- Unit test: mock RPC returns `tx_hex` that deserializes as a valid tx but has trailing garbage bytes → wallet aborts (strict-parse check)
- Unit test: mock RPC returns funding tx with correct txid but understated `value` field in a separate `get_address_utxos` response → wallet uses the value from the deserialized tx, not the RPC field. Transaction builds with correct fee.
- Unit test: generic wallet send where authenticated funding output's script does not match the locally-derived wallet script → abort before signing
- Unit test: HtlcClaim against an output whose script doesn't match the reconstructed locked script → abort before signing
- Unit test: HtlcClaim where the CLI-provided timeout differs from any default — verify the reconstructed locked script uses the CLI timeout
- Unit test: `authenticated_output_lookup` with `index` out of bounds → abort
- Integration test: full spend flow against a local node works unchanged

## Fix 2 — RPC client must cap response body size (LOW)

### The attack
`src/rpc.rs:877` — `rpc_call` does `read_to_end` on the socket, allocating until the body completes. A malicious RPC streams an oversized response and the CLI falls over from memory exhaustion.

### Required fix
- Parse HTTP headers incrementally before reading body.
- Require a `Content-Length` header. Reject responses without one.
- Cap body size at **8 MiB**. Reject responses where `Content-Length` exceeds the cap, and also enforce the cap during the read itself (in case the server lies about `Content-Length`).
- On cap violation, close the connection and return a clear error.

8 MiB is chosen to be comfortably larger than any legitimate RPC response (block-sized payloads are 4 MiB and blocks aren't returned via RPC in a single response in normal flows) while still bounding memory.

### Tests required
- Unit test: response with `Content-Length: 10000000` → rejected before body read
- Unit test: response with `Content-Length: 100` but actual body is 10 MB → read aborts at cap
- Unit test: response with no `Content-Length` → rejected
- Unit test: normal response under cap → works unchanged

## Fix 3 — Peer transport needs in-flight buffer budget (MEDIUM)

### The attack
`src/network/peer.rs:908-952` and `src/types/mod.rs:24,96` — the reader accumulates entire payloads into memory before verifying the frame HMAC. With `MAX_INBOUND_PEERS = 256` and payloads up to 4 MiB, distinct peers can force ~1 GiB of in-flight unverified buffer. Small miners on low-RAM VPS instances die.

### Required fix

We cannot verify HMAC without the full payload (streaming HMAC verification would be a much larger change and is out of scope). What we can do is bound the aggregate.

- Introduce a global semaphore / atomic byte counter for in-flight pre-verification payload bytes. Budget: **128 MiB** total across all peers.
- Before allocating a payload buffer for a frame, attempt to reserve `frame_length` bytes against the budget. If the reservation would exceed the budget, shed the connection (close with a clear log message).
- Per-peer quota: no single peer may hold more than **8 MiB** of in-flight pre-verification payload at once. Enforce in addition to the global budget.
- Release bytes back to the budget when the frame is verified (HMAC passes or fails — either way the buffer is freed).

### Shedding policy
When the global budget is full and a new frame arrives:
- Prefer shedding the peer currently holding the most in-flight bytes
- Never shed peers in our persistent-peer set (if any)
- Log the shed event with peer address and reason

### Tests required
- Unit test: simulate 256 peers each sending a 4 MiB frame simultaneously → at most 128 MiB reserved; excess peers shed cleanly
- Unit test: single peer attempts to hold 16 MiB in flight → rejected at 8 MiB quota
- Unit test: legitimate traffic pattern (peers sending varied small-to-medium frames) unaffected
- Integration test: node under legitimate high load does not shed honest peers

## Release notes tone
Standard Satoshi posture: describe what changed, why, who's affected. No "critical", no "urgent", no marketing around security. Miners and wallet users can read and decide. Example shape:

> v1.4.2 hardens the wallet against malicious RPC endpoints. Previously, wallet spend paths read UTXO values from RPC responses without verifying them against the funding transaction. A malicious RPC could understate values, causing the wallet to build transactions that sign away the difference as miner fees. Wallets now verify every spent UTXO by fetching and authenticating the funding transaction locally. HTLC claim/reclaim flows now verify locked scripts locally. The RPC client enforces a response body size cap. The peer transport enforces a global in-flight buffer budget.
>
> Wallet users spending against non-local RPC endpoints should upgrade before their next spend.
