# v1.7.0 — Fix 2 cold-bootstrap redesign (recent-checkpoint anchor + retry-across-peers)

**Release type:** Design-change release. Replaces the core anchor-selection approach of v1.5.0 Fix 2 (forward-chain tip validation) with a **recent, per-release-bumped checkpoint**, and adds retry-across-peers + bounded concurrency of candidate-tip validations. Keeps v1.6.0 Fix 1 (utility-based eviction) and the v1.5.2 subscriber-race guard unchanged.

**Spec status:** Round 5 — revised after fourth expert review. Round-5 change: (P2) `BootstrapStalled` peer-action is now explicit — coordinator-scope-only abort, remove the active peer from this coordinator's pool, **no identity/IP strike**. Rationale: at 90 s stall the event is an *absence* of progress, not a delivered offence, so it does not meet the existing narrow `should_strike()` bar (strike only on `DeliveredInvalidHeader`). Test 8a corrected to assert `should_strike` returns `false` for this outcome. Do not implement until reviewer signs off on this revised spec.

Earlier rounds (preserved for context): Round 1 was the initial draft. Round 2 tightened cross-peer continuation (P1), disambiguated concurrency (P1), hardened checkpoint ceremony (P2), specified retarget-boundary fixture regeneration (P2), cleaned up performance math units (P3). Round 3 fixed `DeliveredInvalidHeader` no longer killing honest targets (P1), locked rate-limit as per-core with no constant bump (P2), and made `validated_forward` explicitly append-only (P2). Round 4 added the bootstrap-specific coordinator deadline (`BOOTSTRAP_COORDINATOR_DEADLINE_SECS = 300`) and progress-stall rule (`BOOTSTRAP_COORDINATOR_STALL_SECS = 90`) to cap malicious-target slot occupation at ~5 min instead of ~2 h (P1), made the checkpoint-height selection rule deterministic (P3), and fixed stale `test 11` references (P3).

**Empirical verification:** **Non-negotiable.** v1.7.0 cannot ship without a successful cold-bootstrap reproduction on a fresh Mac/residential/unknown-IP node against the production network, completing forward-chain validation and reaching the network tip within a published target (proposed: ≤ 30 min on a modern laptop). The correct test environment is the target end-user environment; a Linux VPS passing the gate does not unblock shipping. **Additionally**: if the fresh-machine gate trips seed-side rate-limit IP strikes or disconnect loops, that is a ship blocker even if the node nominally reaches tip — see Release Gate for the explicit predicate.

## What this fixes

v1.6.0 ships a correct eviction redesign (Fix 1) but **does not cold-bootstrap from a normal user's machine** — fresh Mac on residential internet, seeded against S1/S2/S3, observed on 2026-04-19 stuck at height 0 for 6h55m with repeated log pattern:

```
v1.5.0 Fix 2 forward-chain validation failed for peer [X]: no concurrent validation slot available
v1.5.0 Fix 2 forward-chain validation failed for peer [Y]: fetch timeout
Authenticated peer 82.221.100.201:9333 disconnected (rate limit exceeded) — recording IP strike
```

The issue is **Fix 2 forward-chain tip validation** inherited unchanged from v1.5.0. It is architecturally too slow and too fragile for the target-user environment at the current chain height.

### Diagnosis

At tip ~309,000 and `ASSUME_VALID_HEIGHT = 130,000`, a fresh node's validator must Argon2-verify **179,105 forward headers** in a single attempt, all within one per-peer session, gated by a rate limiter and a single concurrency slot:

- `MAX_CONCURRENT_TIP_VALIDATIONS_BOOTSTRAP = 1` (types/mod.rs:75) — only one peer-validation at a time; every other peer's TipResponse dispatch fails with `NoSlotAvailable`.
- `VALIDATION_ARGON2_PER_CORE_BOOTSTRAP = 10` Argon2/sec (types/mod.rs:85) — **per-core multiplier**. `ValidationRegime::argon2_rate_per_sec()` in `tip_validation.rs:144` computes the bootstrap cap as `num_cpus × VALIDATION_ARGON2_PER_CORE_BOOTSTRAP`, so on an 8-core Mac the effective cap is **80 H/s**, and on a 16-core desktop it's 160 H/s. At 80 H/s, 179,105 forward headers take **~37 minutes of rate-limited Argon2**, not 5 hours — the rate limit is not the primary cold-bootstrap bottleneck.
- `TIP_VALIDATION_BATCH_TIMEOUT_SECS = 120` per GetHeaders (types/mod.rs:89) — during the ~5-hour wall-clock the TCP connection goes idle; seeds close it; the next batch request times out and the whole attempt is discarded.
- **No resumable state.** Every retry re-fetches the checkpoint header, re-fetches the 4,319-header pre-checkpoint retarget lookback, re-starts forward validation at `anchor+1`. A 5-hour attempt that fails on batch 2,700 of 2,800 loses all its work.
- **Seeds strike the fresh IP.** Seeds rate-limit the fresh residential IP after enough retry attempts, causing disconnection loops that push us further off the reliable-peer set.

These symptoms compound. The fundamental scaling problem is the **length of the single-peer validation session** (~37 min even at the current rate cap, on a typical 8-core laptop). At that length, the session almost always trips a transport-level failure — seeds strike the unfamiliar IP, TCP idle timeout kills the connection mid-compute, or the peer's per-connection budget closes us — and since the round-1 code has no resume, every failure restarts from the anchor. The 5K-header window in Change 1 + retry-across-peers in Change 2 together collapse the session length to ~1 min on the same laptop, well below any reasonable TCP-idle or rate-limit-strike window.

### Why this is a release blocker, not a known limitation

Exfer is a decentralized blockchain. The target user runs a node from a personal machine on a home connection from an IP unknown to the seed set. That IS the representative environment. If v1.6.0 ships and this user cannot advance past genesis in a reasonable time, the software is not serving its primary purpose.

## Design

Three changes, in descending order of importance. Change 1 alone is expected to move the needle from "cannot bootstrap" to "bootstraps in minutes"; changes 2 and 3 harden against residual failure modes.

### Change 1 (primary): Per-release recent-checkpoint anchor

**Bump `ASSUME_VALID_HEIGHT` / `ASSUME_VALID_HASH` / `ASSUME_VALID_CUMULATIVE_WORK` to the highest retarget-boundary height `≤ tip_at_release − 5000`.**

`ASSUME_VALID_HEIGHT` must be a retarget-boundary height (multiple of `RETARGET_WINDOW = 4320`) for the pre-checkpoint retarget-lookback math to stay clean. Selection rule: compute `tip_at_release − 5000`, round **down** to the nearest multiple of 4320. This puts the anchor in the range `[tip − 9319, tip − 5000]` — never less than 5000 behind tip (so fresh nodes don't accidentally set the anchor too close to tip during a short-window release), and never more than ~9320 behind (so the forward span stays under one retarget window's worth of headers).

Today those constants point at h=130,000. At current tip ~310,000 the round-4 rule would select `h = 302,400` (= 70 × 4320), which lands 7,600 blocks behind tip. The fresh node forward-validates ~7,600 headers — about **~1.5 minutes** on a typical 8-core laptop (7,600 / 80 H/s = 95 s at the per-core rate cap), vs. ~37 min at the current 180K-header span. A single-peer session that short is comfortably below every transport-level failure threshold we've seen. 7,600 is also well under the round-4 Change-4 deadline of 300 s.

**This is the same technique Bitcoin Core uses with `chainparams.assumevalid`.** Every Core release ships with an `assumevalid` hash updated to a block that was recent at build time. Fresh nodes trust the contained-in-binary constant transitively; they still verify PoW for every block *after* the assumevalid point, which is the window that actually needs attacker resistance (reorgs deeper than a few days are infeasible to manufacture against a well-worked chain).

**Trust model is unchanged from today's design.** We already trust `ASSUME_VALID_HASH`. Moving the constant closer to tip does not weaken it — an attacker who could forge a fake checkpoint at h=305,000 could equally forge one at h=130,000; the cryptographic assumption is the same.

**Per-release ceremony — multi-source verification required.** Single-source checkpoint pulls are too trusting for a value that becomes a trusted release anchor. Procedure:

1. **Pick target height deterministically.** `target_height = ((tip_at_release − 5000) / RETARGET_WINDOW) × RETARGET_WINDOW`, using integer division. This is the highest retarget-boundary height at or below `tip − 5000`. Guarantees: `target_height ≤ tip − 5000` and `tip − target_height < tip − 5000 + RETARGET_WINDOW` (i.e., `< 9320` blocks behind tip).
2. **Query at least two independent canonical nodes** for block at the target height. Acceptable combinations:
   - S1 + S2, S2 + S3, S1 + S3 (any 2 of our in-house nodes, if all are healthy), OR
   - 1 in-house node + 1 external full-node peer the release manager personally controls or trusts, OR
   - 3+ in-house + external sources if healthy nodes are limited.
3. **Required agreement:** each queried source MUST return the same `(block_id, cumulative_work)` tuple for the target height. Any disagreement aborts the release and triggers investigation (compromised node, RPC bug, reorg straddling query window).
4. Record the agreed `(height, block_id, cumulative_work)` triple. Update the three constants in `src/types/mod.rs`.
5. **Regenerate the retarget-boundary fixture** for `assume_valid_cumulative_work_guard.rs` (see below). This is mandatory, not optional.
6. Build, run full test suite, confirm the regenerated guard passes.
7. Proceed with release.

**Retarget-boundary fixture regeneration.** The existing `assume_valid_cumulative_work_matches_fixture_recomputation` test (`tests/assume_valid_cumulative_work_guard.rs:95`) holds a hardcoded list of `(retarget_window_start_height, difficulty_target_hex)` entries spanning from genesis to `ASSUME_VALID_HEIGHT` and recomputes the cumulative work from them to cross-check the baked-in constant. Bumping `ASSUME_VALID_HEIGHT` requires extending this fixture with the new retarget-window entries between the old checkpoint and the new one; the test's "31 entries" comment and source-timestamp need updating each release. The regeneration helper (a small script that queries the same multi-source canonical nodes for each retarget-window start height's difficulty_target) should be committed alongside the test so the ceremony is reproducible. Treat this as a required release-prep artifact, not an implementation-time afterthought.

**Cadence.** Every release for now (expert steer). Once the ceremony is boring and reliable, migrate to a "max forward-span" rule — e.g., bump only if `tip - ASSUME_VALID_HEIGHT > 50,000` — so we don't churn constants in point releases with no chain growth.

**Per-release ceremony only affects new installs.** Nodes already in the field on older checkpoints continue to sync from their stored chain; the bumped constant only matters for newly-installed binaries. Confirmed spec-level: startup/runtime uses the constant only as a trust anchor for first-IBD Argon2 gating and checkpoint-header verification, never to reject or reorg against already-validated blocks in storage (to be verified in the implementation diff).

### Change 2 (secondary): Retry the forward-chain span across peers, against a **fixed target tip**

Today, if peer A delivers 2,500 of 5,000 expected headers and then times out or disconnects, the whole attempt aborts and peer B starts over from the anchor. Change 2 preserves `validated_forward` state across the bootstrap attempt (not across process restarts) and resumes against a different peer on transport-level failures — but **only if the new peer is committing to the same target tip**.

**Target model.** A bootstrap coordinator instance is parameterised by a *single fixed target*: `TargetTip = (height, block_id, cumulative_work)`. The target is selected **once**, at coordinator start, from the best peer-advertised tip observed at that moment. The target does NOT follow subsequent tip advertisements from other peers — those are different candidates and get their own coordinator instance (see Change 3 for how many can run concurrently).

**Cross-peer continuation is valid iff:**

1. The new peer's delivered header batch `prev_block_id` field links to the *last validated header* in our accumulated `validated_forward` (or to the anchor if `validated_forward` is empty). Linkage is SHA-chain exact match, not a range check.
2. The new peer's tip claim (as advertised via the TipResponse / GetHeaders protocol) is the same `(height, block_id, cumulative_work)` as our `TargetTip`. A peer claiming a different tip is not a valid continuation source for this coordinator, even if their first delivered header happens to link to our prefix — they may be on a fork that diverges later.
3. The final validated header in the chain, once `validated_forward.len() == target_height - anchor_height`, MUST have `block_id == TargetTip.block_id` and the accumulated work MUST equal `TargetTip.cumulative_work`. Mismatch is a **terminal failure of the coordinator** — either the peer lied about the target from the start (invalid-header class, strike) or we accepted a prefix from a peer whose claim was valid at the moment of selection but then got reorged out (target-orphaned class, no strike, abort the coordinator).

**Error classification.** `validated_forward` is **append-only**: a header is committed to it only after its PoW, difficulty (via overlay), and parent-linkage check all pass. Once committed, it is not rolled back short of coordinator abort. This append-only invariant means all recovery paths simply pick up at `anchor + validated_forward.len()` — no bookkeeping for "which prefix is still trusted."

| Outcome | Peer action | Coordinator action | State change |
|---|---|---|---|
| `FetchTimeout`, `PeerDisconnected`, `NoSlotAvailable` | mark peer stalled for this coordinator (not a strike — could be network/peer-overload) | pick another peer from `peer_pool` advertising the same `TargetTip`; resume at `anchor + validated_forward.len()` | no change |
| Peer advertises a different tip than our `TargetTip` | not a strike — peer is on a different (possibly honest) fork; exclude from this coordinator's pool | pick another peer advertising `TargetTip`; if pool empty within deadline, abort (target-unreachable) | no change |
| `DeliveredInvalidHeader` in a batch — structurally bad (wrong height, failed Argon2, bad prev-link within the batch) | **strike the peer**, remove from `peer_pool` | **continue with another peer advertising the same `TargetTip`**. A liar does not kill the target — the honest peers can still complete validation. | pre-break portion of the batch (if any headers in the batch passed before the bad one) is committed; post-break portion and any tentatively-parsed work discarded. `validated_forward` grows by the pre-break count only |
| Same as above but **first** header of the batch fails to link to our prefix | strike the peer (they delivered a chain inconsistent with the prefix the rest of the pool is advertising for this target) | continue with another peer | `validated_forward` unchanged (nothing committed from this batch) |
| Final `validated_forward` reaches `target_height` but last-header `block_id` ≠ `TargetTip.block_id` OR accumulated work ≠ `TargetTip.cumulative_work` | **strike the peer who delivered the final divergent batch** — they lied about completing to TargetTip (all honest target-peers would converge to the same final header) | abort — target is dead | state discarded |
| `peer_pool` becomes empty (all peers struck, stalled-past-deadline, or moved to a different tip) before target reached | none | abort (target-unreachable) — other coordinators on other `TargetTip`s may still succeed | state discarded |
| `BootstrapStalled` — 90 s elapsed since last `validated_forward` progress, regardless of per-batch timeouts (Change 4) | **remove active peer from this coordinator's `peer_pool` only; DO NOT record an identity/IP strike** (see rationale below) | abort (this coordinator) | state discarded |
| Overall `BOOTSTRAP_COORDINATOR_DEADLINE_SECS = 300` wall-clock exceeded (Change 4) | none | abort | state discarded |

**Key round-3 tightening (per expert P1):** a single malicious peer who joins an honest target's pool and feeds one bad batch costs themselves (strike + pool removal) but **does not kill the target**. The honest peers remaining in the pool can continue validation from where the prefix left off. Only final-tip mismatch or full pool exhaustion kills a target. This eliminates the cheap liveness-DoS attack surface the round-2 spec left open.

**Key round-3 tightening (per expert P2, linkage):** `validated_forward` is append-only. A mid-batch linkage failure commits the headers that passed, discards the rest of that batch, and retries with a different peer at `anchor + validated_forward.len()`. There is no scenario where committed headers are rolled back short of coordinator abort. The round-2 "discarded back to anchor + min(...)" phrasing is removed.

**Key round-5 tightening (per expert P2, BootstrapStalled peer-action):** `BootstrapStalled` is a **coordinator-scope-only** abort, not a strike-triggering condition. Rationale:

- At the moment the 90 s stall timer fires, no single identifiable event has occurred — only an *absence* of progress across some sequence of peer interactions. Attributing that absence to whichever peer happens to have been "last active" would penalise an identity based on timing rather than on a concrete delivered offence.
- The existing narrow strike policy (`should_strike()` at `tip_validation.rs:497`) deliberately only strikes on `DeliveredInvalidHeader` — a peer must have provably delivered a bad header to earn an identity/IP strike. `BootstrapStalled` does not meet that bar.
- Round-4's stall bound does its job without strikes: when the coordinator aborts, the slot frees and the active peer is removed from *this* coordinator's `peer_pool` (moot since the coordinator dies). The peer can legitimately participate in a future coordinator or a different `TargetTip`'s pool; it simply doesn't help this run.
- The adversary still pays a real cost: they wasted a slot for up to 90 s and got nothing out of it. They cannot "ban" a target or eclipse the honest network by running stall attacks — at most they delay one coordinator cycle, which is already re-tried automatically by the sync manager on the next TipResponse re-advertisement.

`should_strike()`'s narrow policy therefore remains correct as-is; no change is needed to the strike classifier for Change 4.

**Concretely:**

- Move the forward-fetch loop out of the per-peer `run_tip_forward_validation` function and into a sync-manager-level "bootstrap validation coordinator" that holds:
  - `target_tip: TargetTip` (immutable for the coordinator's lifetime)
  - `validated_forward: Vec<BlockHeader>` (append-only except on abort)
  - `peer_pool: HashSet<PeerId>` (peers currently advertising `target_tip`)
  - `stalled: HashMap<PeerId, Instant>` (temporarily-skipped peers with a backoff)
  - `deadline: Instant` (overall coordinator wall-clock)
- Coordinator picks a peer from `peer_pool \ stalled`, asks for the next batch from `anchor + validated_forward.len()`, validates the batch as before (PoW, difficulty via overlay, linkage to prefix).
- On the outcomes above, apply the table's action.
- A coordinator's lifetime ends on: target reached (success), target-unreachable (no peers left, or all stalled past deadline), target-divergent (last-header mismatch — strike), or wall-clock deadline.

This change is self-contained to `src/network/sync.rs`; it adds a coordinator struct but does not change any public API or protocol.

### Change 3 (tertiary): Bounded concurrency of **distinct-target** coordinators

Bump `MAX_CONCURRENT_TIP_VALIDATIONS_BOOTSTRAP` from 1 to 4. **The semantics: up to 4 simultaneous bootstrap coordinator instances, each validating a *distinct* `TargetTip`.** This is the model the round-2 spec locks in; interpreting the bump as anything else (multiple peer lanes inside one coordinator, or four redundant coordinators on the same target) is explicitly out of scope and would introduce the problems the expert flagged (duplicate Argon2, races in shared state, redundant revalidation).

**Precisely:**

- Within one coordinator: **peer fetch is serial**. The coordinator asks one peer for the next batch, waits, validates, advances `validated_forward`, asks the *same or a different* peer for the next batch. No parallel fetch lanes inside a coordinator — this would race on `validated_forward` and gain nothing (Argon2 is the bottleneck, not fetch RTT).
- Across coordinators: **up to 4 coordinator instances may run simultaneously, each on a different `TargetTip`**. This is how the `NoSlotAvailable` cascade is quieted — when peer B advertises a different tip than peer A, peer B gets its own coordinator instead of being turned away.
- **Dedup rule:** if a new peer's TipResponse claims a `TargetTip` already owned by an active coordinator, the peer is added to that coordinator's `peer_pool` rather than spawning a redundant coordinator. Two coordinators MUST NOT exist for the same `(height, block_id, cumulative_work)` triple.
- **Eviction rule when 4 are full and a 5th target arrives:** the 5th TipResponse is dropped (quietly — no log spam, no strike) and can be re-considered when a coordinator slot frees. Acceptable because peers re-advertise tips on ping/pong cycles.
- **Rate limiter is shared across all 4 coordinators** at the existing `num_cpus × VALIDATION_ARGON2_PER_CORE_BOOTSTRAP` rate (per-core semantics preserved; see Rate-Limit Model below). The point of 4 is peer-diversity and stall-tolerance, not throughput multiplication.

**Rate-limit model is explicitly preserved, not changed.** Round-2 proposed raising `VALIDATION_ARGON2_PER_CORE_BOOTSTRAP` from 10 to 40 on the assumption that the limiter was the primary bottleneck. That assumption was wrong — per-core semantics means the current per-laptop cap is already 80 H/s (on 8 cores), which processes a post-Change-1 5K-header span in ~1 minute. **No bootstrap rate-limit constant change is proposed for v1.7.0.** The `num_cpus × VALIDATION_ARGON2_PER_CORE_BOOTSTRAP` formula in `tip_validation.rs:144` stays as-is. The expert-flagged round-2 inconsistency (treating the cap as a shared total) is resolved by keeping the per-core model and recalibrating the numbers throughout this spec.

### Change 4 (round-4 addition): Tight bootstrap coordinator deadline + progress rule

With Change 3 permitting 4 concurrent coordinator slots, an adversary who advertises 4 distinct malicious `TargetTip`s can squat all 4 slots if each slot's deadline is large. The current `compute_deadline()` formula (`tip_validation.rs:377`) uses `max(TIP_VALIDATION_DEADLINE_FLOOR_SECS = 7200, ceil(expected_argon2_secs × 1.5))` — a **2-hour floor** regardless of how small `expected_headers` is after Change 1. A malicious peer replying just under the 120 s batch timeout holds the slot for the full 7200 s, blocking honest 5th-target candidates for two hours per attack cycle.

Round-4 fix — **two independent abort predicates**:

1. **Hard wall-clock deadline (low-minutes).** Introduce a new constant:
   ```rust
   pub const BOOTSTRAP_COORDINATOR_DEADLINE_SECS: u64 = 300; // 5 min
   ```
   The bootstrap regime branch of `compute_deadline()` uses this constant as its floor instead of `TIP_VALIDATION_DEADLINE_FLOOR_SECS`. The `ceil(expected_argon2_secs × 1.5)` formula continues to apply; if it's larger than 300 s for some future larger-span bootstrap, it wins. Sanity check: post-Change-1 5K headers / 80 H/s × 1.5 = 94 s, so 300 s dominates and gives a 3× safety margin over honest completion. Steady-state regime keeps the 7200 s floor unchanged.

2. **Progress-based abort.** A coordinator aborts early if it fails to make forward progress on `validated_forward.len()` within a stall window, regardless of whether individual batches are timing out:
   ```rust
   pub const BOOTSTRAP_COORDINATOR_STALL_SECS: u64 = 90; // 90 s
   ```
   If `validated_forward.len()` has not increased in the last 90 s, the coordinator aborts with `BootstrapStalled`. This catches the "peer replies exactly every 119 s with an empty or prefix-failing batch" pattern that exploits per-batch timeouts while never advancing the prefix.

Combined, a malicious target can occupy a coordinator slot for at most `min(BOOTSTRAP_COORDINATOR_DEADLINE_SECS, BOOTSTRAP_COORDINATOR_STALL_SECS_after_last_progress)` — roughly **5 min worst case, 90 s if no progress at all**. Four such attacks together block the 5th-target slot for at most ~5 min per cycle, not 2 h.

**What this costs an honest target:** the 300 s deadline gives a 3× margin over the ~62 s honest compute time on an 8-core laptop, and ~6× on a 16-core desktop. Pathologically slow machines (old dual-core laptop doing Argon2id at 20 H/s) take 250 s for 5K headers — still under the 300 s deadline. If a machine is so slow that it genuinely needs more than 5 min for a 5K-header span, it will fail the release gate on its own, which is the correct outcome (the release gate exists to catch exactly this class of machine).

**What this costs an honest target with a flaky peer:** each peer-switch does not reset the coordinator wall-clock. But as long as *some* peer advances the prefix within 90 s windows, the coordinator runs until target or the 300 s cap. The prefix is append-only (Change 2, round-3), so every bit of progress is preserved across peer swaps.

**Early-abort + slot rotation.** When a coordinator aborts (stall, deadline, or any other exit class), the slot frees immediately. Queued 5th-target TipResponses are not persisted (no queue), but peers re-advertise tips on ping/pong cycles, so a freed slot picks up a new target on the next re-advertisement. No explicit queue complexity added.

## Test plan

Unit + integration tests:

1. **`assume_valid_constants_guard_after_checkpoint_bump`** — existing release-hardening test passes with the bumped constants *and* the regenerated retarget-boundary fixture. Verify by running the regen helper against two canonical sources, confirming the fixture entries agree across sources, and running the guard.
2. **`bootstrap_coordinator_retries_on_fetch_timeout_same_target`** — coordinator with `TargetTip = T`, peer A delivers 2 batches linking to prefix then stalls. Peer B (also advertising T) takes over; `validated_forward` is kept; coordinator completes against T.
3. **`bootstrap_coordinator_rejects_different_tip_peer`** — coordinator with target T. Peer B advertises tip T' ≠ T and attempts to serve a batch. Coordinator must NOT accept B's headers as continuation, MUST NOT strike B (different fork, not adversarial), MUST keep `validated_forward` and try another T-advertising peer.
4. **`bootstrap_coordinator_strikes_bad_peer_but_keeps_prefix`** — peer A delivers 3 valid batches (192 headers committed to `validated_forward`), then peer B joins the pool advertising same `TargetTip` and delivers a batch with one bad-PoW header. **Assert:** peer B is struck and removed from pool; `validated_forward` retains all 192 pre-B-attempt headers; post-break portion of B's bad batch is dropped; coordinator continues against peer A (or another same-target peer) at `anchor + 192`, eventually reaching target. **This is the round-3 P1 invariant — one liar cannot kill an honest target.**
5. **`bootstrap_coordinator_validated_forward_is_append_only`** — across a sequence of (successful batch, failed batch from different peer, successful batch) operations, `validated_forward.len()` is monotonically non-decreasing. Prior committed headers are never retroactively rolled back short of coordinator abort. **This is the round-3 P2 linkage invariant.**
6. **`bootstrap_coordinator_detects_final_tip_mismatch`** — prefix validates end-to-end but the final header's `block_id` ≠ `TargetTip.block_id`. The peer delivering the divergent final header is struck; coordinator aborts.
7. **`bootstrap_coordinator_aborts_on_pool_exhaustion`** — all peers advertising `TargetTip` are struck, stalled, or move off-target; coordinator aborts (target-unreachable) cleanly; state discarded; other coordinators on other targets are unaffected.
8. **`bootstrap_coordinator_respects_overall_deadline`** — hits the `BOOTSTRAP_COORDINATOR_DEADLINE_SECS = 300` wall-clock and aborts, independent of per-batch timeout; `validated_forward` discarded cleanly.
8a. **`bootstrap_coordinator_aborts_on_progress_stall`** (Change 4, new) — peer replies to each GetHeaders batch at ~110 s (under the 120 s FetchTimeout) with headers that fail prefix linkage, so `validated_forward.len()` never increases. Coordinator must abort with `BootstrapStalled` at the `BOOTSTRAP_COORDINATOR_STALL_SECS = 90` mark, well before the 300 s overall deadline. **Assert the active peer receives no identity/IP strike** — `should_strike(&Err(BootstrapStalled))` returns `false`, matching the existing narrow policy. The slot frees for a new coordinator.
8b. **`four_bogus_coordinators_free_slots_within_five_minutes`** (Change 4, new) — fire 4 distinct bogus `TargetTip`s simultaneously, each backed by a peer that plays the stall-just-under-120s game. **Assert:** within `BOOTSTRAP_COORDINATOR_DEADLINE_SECS = 300` of the first coordinator start, all 4 slots have aborted and at least one is free for an honest 5th target. This is the regression test for the round-3-to-round-4 fix.
9. **`concurrent_coordinators_dedupe_same_target`** — when peer C advertises a tip already owned by an active coordinator, peer C is added to that coordinator's `peer_pool` rather than spawning a second coordinator on the same target. Assert active coordinator count stays ≤ 4 and no two have equal `TargetTip`.
10. **`concurrent_coordinators_on_distinct_targets_share_rate_limiter`** — 4 coordinators simultaneously active on 4 different `TargetTip`s. Aggregate Argon2 throughput bounded by the `num_cpus × VALIDATION_ARGON2_PER_CORE_BOOTSTRAP` per-core cap (not multiplied by 4). No state races between coordinators (each owns its own `validated_forward`).
11. **`fifth_target_is_dropped_silently`** — with 4 coordinators full, a 5th distinct TipResponse arrives. It is silently dropped (no log spam, no strike). When a coordinator frees, a subsequent re-advertisement of the same tip spawns a coordinator normally.
12. **`no_slot_available_log_cascade_is_quieted`** — the existing `NoSlotAvailable` warning is scoped to the actual-problem case ("dispatched but failed") and suppressed for the quiet-queue case ("no slot; try later"). Verified by counting warnings under a synthetic load.

Empirical (release gate):

13. **`coldboot_fresh_mac_residential_reaches_tip_cleanly`** — on a fresh Mac from a residential connection, with an empty datadir, seeded only with S1/S2/S3, the node completes forward-chain validation and reaches tip within 30 min wall-clock **without tripping seed-side IP strikes or rate-limit disconnect loops**. Reproduction script committed alongside the release so the gate is deterministic. The "clean" predicate is checked by grepping the node's run log and the seeds' logs for strike events attributable to the test IP during the run.

## Release gate

**Non-negotiable.** v1.7.0 cannot be tagged / pushed / announced until:

1. All unit + integration tests above pass on CI.
2. The empirical gate (test 13) has been executed on a fresh Mac from a residential connection, seeded against the production network, reached tip within the published target, **and produced zero seed-side rate-limit IP strikes attributable to the test IP during the run**. A strike even if the node nominally reaches tip = ship blocker, because it indicates the seed-side rate-limit issue is still present and a different-seed constellation might not be so lucky. (The spec leaves seed-side rate-limit code changes out of scope for v1.7.0, but the release gate checks the symptom.)
3. The Linux-x86_64 counterpart of the empirical gate has also passed (in addition, not instead of Mac).
4. Expert review of the implementation diff has signed off after the code is written.

"Works in the Linux datacenter" is **not** a passing grade. The cold-bootstrap failure that this release exists to fix was observed on a Mac/residential machine; the same environment must be used to verify the fix.

## Resolved from round 1 (expert steer)

All of the round-1 open questions have been resolved and folded into the body above:

1. **Cadence:** every release for now. Migrate to "max forward-span" rule (e.g., bump only if `tip - ASSUME_VALID_HEIGHT > 50K`) once the ceremony is boring. See Change 1 Cadence paragraph.
2. **Anchor bump for already-synced nodes:** spec-level safe — the constant is only a trust anchor for first-IBD Argon2 gating and checkpoint-header verification, not for rejecting or reorging against stored blocks. Implementation-diff reviewer must confirm this holds in code.
3. **Seed rate-limit:** code changes remain out of scope for v1.7.0, but the symptom is in scope for the release gate (test 13). A fresh-machine run that trips seed-side IP strikes is a ship blocker even if the node nominally reaches tip.
4. **Coordinator state lifetime:** no persistence across process restarts for v1.7.0. With the checkpoint near tip, the forward span is short enough that on-disk resumability is unnecessary. Revisit if a future release grows the span for any reason.

## Non-goals

- Consensus-level change — none needed, this is a sync-path performance / UX fix.
- Protocol version bump — none, wire format unchanged.
- Per-header Argon2 removal — Argon2 verification of every forward header post-assumevalid remains the honest path to catching a dishonest peer; we are not weakening it, only shortening the span.
