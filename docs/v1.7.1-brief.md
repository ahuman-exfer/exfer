# v1.7.1 — Cold-bootstrap fix: BlockResponse rate-limit exempts pre-confirmation bootstrap

**Release type:** Point release. Narrow bug fix on top of v1.7.0's unshipped code. The `MAX_BLOCKS_PER_MIN = 12` cap in the BlockResponse handler exempts only the active IBD peer, not the broader pre-confirmation bootstrap window. Before the sync manager has ever confirmed a peer via Fix 2, orphan-parent chasing produces a cascade of BlockResponse messages from honest seeds that strikes them off the peer set and breaks cold-bootstrap entirely.

**Scope:** One new `Node` field (sticky `AtomicBool`) + one modified reader-loop branch (BlockResponse handler). No constant changes. No protocol change. No consensus change. No touch to Fix 2's forward-chain validator.

**Spec status:** Round 4 — revised after round-3 review surfaced that `best_peer_work == [0; 32]` is **not** the sticky "ever confirmed a peer" invariant round-3 claimed it was. The field is recomputed per sync-manager loop from currently-connected confirmed peers (`src/network/sync.rs:5241`, `:5273`, written back at `:5457`), so if all confirmed peers disconnect mid-run, the field goes back to zero and round-3's exemption would accidentally re-enable for a fully-synced node. Round-4 replaces the signal with a **new sticky `ever_confirmed_peer: AtomicBool` flag** set once on first confirmation and never reset within a process lifetime. Round-4 also **drops Change B** — round-3's aggregate-counter justification for bumping `MAX_BLOCKS_PER_MIN` was wrong (`block_count` is per-peer-reader-local, not shared across peers per `src/network/sync.rs:3543`), and there is no per-peer rationale to substantiate the bump separately. v1.7.1 is now a single-change release.

Earlier rounds preserved for context: Round 1 had the root-cause story wrong in direction (assumed Live startup, assumed NewBlock fires). Round 2 corrected to "fresh node starts in CatchingUp; NewBlock is exempt there; BlockResponse fires the strike via orphan-parent chase because its exemption is only `is_ibd_peer`." Round 3 broadened Change A's signal from `SyncState::CatchingUp` to `best_peer_work == [0;32]` and committed to a Change B constant bump; round 4 corrects the signal choice and drops the bump.

**Empirical verification:** Same non-negotiable fresh-Mac / residential / unknown-IP gate as v1.7.0 test 13. Target: ≤ 30 min wall-clock to tip, **zero self-inflicted strikes against S1/S2/S3** during the run.

## What this fixes

v1.7.0's release gate (test 13) failed on 2026-04-20 after 17 minutes:
- final height: 0 (genesis only)
- 9 strikes against S2 (82.221.100.201) — all logged as `Authenticated peer ... disconnected (rate limit exceeded)`
- 0 successful Fix 2 validations

Round-1 of this spec attributed the strikes to `MAX_BLOCKS_PER_MIN = 12` firing in the NewBlock handler. That diagnosis was wrong. This round traces the actual path.

---

## Correct state-machine story

A fresh node initializes `sync_state = SyncState::CatchingUp` (`src/main.rs:3517`), not `Live`. The sync manager flips it to `Live` only when either:

- `has_confirmed_peer == true` and the node is not-too-far-behind (`src/network/sync.rs:5463-5465`), or
- `connected_count == 0 && !ever_had_peer && start_time.elapsed() > 60s` (no peers for a minute — the "nobody's home" fallback at `src/network/sync.rs:5466-5470`).

During the Fix 2 pre-confirmation window (no peer has been confirmed yet, but we do have inbound/outbound connections), the node is in `CatchingUp`. The NewBlock handler's `catching_up` guard at `src/network/sync.rs:3617-3625` correctly skips its rate limit under this state. **NewBlock is not the cap that fires.**

## What actually fires

Inbound `NewBlock` delivery routes into `process_block_event` (`src/network/sync.rs:4455+`). The fresh node has only genesis, so every broadcast block's parent is unknown. The handler adds the block to the orphan buffer and (`src/network/sync.rs:4498-4503`):

```rust
if should_request {
    node.send_to_peer(&from_identity, Message::GetBlocks(vec![parent_hash])).await;
}
```

The seed responds with `Message::BlockResponse(parent_block)`. Our reader loop lands in the BlockResponse handler at `src/network/sync.rs:3873-3885`:

```rust
Message::BlockResponse(block) => {
    let is_ibd_peer = {
        let guard = self.active_ibd_peer.lock()...;
        guard.map_or(false, |(id, _)| id == meta.identity)
    };
    if !is_ibd_peer {
        block_count += 1;
        if block_count > MAX_BLOCKS_PER_MIN {
            return Err(PeerError::RateLimitExceeded);
        }
    }
```

The exemption here is **only** `is_ibd_peer` — the single peer selected by the sync manager as the active IBD peer. Since Fix 2 pre-confirmation precedes any IBD-peer selection, `is_ibd_peer == false` for every peer, so every `BlockResponse` increments `block_count`. The `block_count` counter is **shared** with the NewBlock handler (same reader-loop-local `mut` variable, reset every 60 s), and the cap is `MAX_BLOCKS_PER_MIN = 12`.

Each new network block broadcast to us triggers an orphan-parent chain walk: we receive NewBlock(N), request parent N-1, seed delivers BlockResponse(N-1); N-1 is also orphan, we request N-2, etc. A typical orphan chase during an 80-second window at ~9 blocks/min of live network produces well over 12 `BlockResponse` messages from a single seed. After the 13th we strike and tear down the session, which also kills any Fix 2 validation mid-flight on that peer. Reconnect, window resets, cycle repeats. The observed 9-strikes-in-17-min pattern matches this cleanly.

## The asymmetry

- `Message::NewBlock`: rate-limited unless `catching_up` — **does** honor the CatchingUp exemption.
- `Message::BlockResponse`: rate-limited unless `is_ibd_peer` — does **not** honor CatchingUp.

Both paths feed into the same block-processing pipeline and the same orphan-chase behavior. The exemption asymmetry was the root cause all along.

## Design

**Single change.** Earlier drafts bundled a second "defense-in-depth" constant bump; round-4 drops it — see "Why Change B is dropped" below.

### Change A: BlockResponse exempts pre-confirmation bootstrap via a sticky process-lifetime flag

Add a new sticky `AtomicBool` to `Node`:

```rust
pub ever_confirmed_peer: std::sync::atomic::AtomicBool,
```

Semantics: initialised to `false` on `Node::new()`. **Set to `true` at every site in the codebase that writes `PeerTip { ..., confirmed: true }` under a same-session guard.** At time of writing there are exactly two such sites in `src/network/sync.rs`:

- `:5703` — the async forward-chain validation task's `Ok(vt)` handler (v1.5.0 Fix 2 path).
- `:5800` — the legacy single-header validated-tip confirmation path.

At both, after the `lp.tip = Some(PeerTip { ..., confirmed: true })` write, add `self.ever_confirmed_peer.store(true, Ordering::Relaxed)`. Since the flag is sticky, the second-through-Nth writes are idempotent no-ops; no CAS needed. **Never cleared** within a process lifetime. A process restart resets it to `false`, which is fine: a just-started process with prior chain state on disk is by construction in a "could plausibly need orphan chase" window anyway.

Grep invariant to enforce at review time: every occurrence of `confirmed: true` in a `PeerTip { ... }` struct literal must be accompanied by an `ever_confirmed_peer.store(true, ...)` call in the same block. If a future change adds a new confirmation site, the reviewer must check this invariant holds.

Broaden the BlockResponse handler's exemption from `is_ibd_peer` only to `is_ibd_peer || !ever_confirmed_peer`:

```rust
Message::BlockResponse(block) => {
    let is_ibd_peer = {
        let guard = self.active_ibd_peer.lock()...;
        guard.map_or(false, |(id, _)| id == meta.identity)
    };
    // v1.7.1 Change A: pre-confirmation bootstrap exemption.
    // Sticky AtomicBool, set once by the sync manager on first
    // Fix 2 confirmation, never reset within a process lifetime.
    // Covers every pre-confirmation state — CatchingUp initial,
    // the 60s-no-peers fallback to Live (src/network/sync.rs:5466),
    // and early post-restart windows before re-confirmation — during
    // which honest orphan-parent chasing can produce BlockResponse
    // cascades that would otherwise strike the seed off our peer set.
    let pre_confirmation_bootstrap =
        !self.ever_confirmed_peer.load(std::sync::atomic::Ordering::Relaxed);
    if !is_ibd_peer && !pre_confirmation_bootstrap {
        block_count += 1;
        if block_count > MAX_BLOCKS_PER_MIN {
            return Err(PeerError::RateLimitExceeded);
        }
    }
    // rest unchanged
```

**Why a sticky flag, not `best_peer_work == [0; 32]`.**

Round-3 proposed the zero-work check, but `best_peer_work` is a running aggregate of currently-connected confirmed peers, recomputed each sync-manager loop iteration (`src/network/sync.rs:5241`, `:5273`, `:5457`). If every confirmed peer disconnects — due to network flap, restart of upstream seeds, or just natural peer churn — the value goes back to zero within one loop iteration. A fully-synced node in that state would accidentally re-enter the exemption and have its BlockResponse cap disabled, which is wrong. The field's doc comment describes it as "best confirmed peer cumulative work," not a ever-confirmed marker. Using it as one was a spec mistake.

A dedicated `AtomicBool` with explicit "set once, never reset" semantics closes the invariant cleanly:

| State | `ever_confirmed_peer` | `is_ibd_peer` | Exemption? |
|---|---|---|---|
| Fresh node, CatchingUp, no peers connected yet | `false` | `false` | ✅ |
| Fresh node, CatchingUp, peers connected, Fix 2 not done | `false` | `false` | ✅ |
| Fresh node, 60s fallback to Live, peers then arrive | `false` | `false` | ✅ (the round-3 edge case — now correctly handled) |
| Fix 2 confirmed a peer → sync manager selects IBD peer | `true` (set once here) | `true` | ✅ |
| Synced & running, steady state | `true` | `false` | ❌ (rate limit active — correct) |
| **Synced, all confirmed peers disconnect** | `true` (sticky) | `false` | ❌ (correct — the round-3 bug is fixed here) |
| Synced, sync manager re-entered CatchingUp from Live | `true` | `false` | ❌ (rate limit active; orphan chase should be short) |
| Process restart of a previously-synced node | `false` (reset on restart) | `false` | ✅ (correct — a just-restarted node may need to catch up) |

The second-from-last row is what round-3 got wrong — round-3 would exempt here too, disabling the cap more broadly than intended. Round-4's sticky flag holds `true` through peer churn.

**Alternative considered: tip-height gate.** A signal like `tip.read().height == 0 && !ever_confirmed_peer` would express "truly fresh" more narrowly, but requires an async read on every BlockResponse. The sticky flag is cheaper (atomic relaxed load) and semantically sufficient — if the process has confirmed a peer at any point, the orphan-chase cascade it might now be seeing is downstream of chain state it deliberately has, and can be rate-limited honestly.

### Why Change B is dropped

Round-2 and round-3 proposed bumping `MAX_BLOCKS_PER_MIN` from 12 to 60 as defense-in-depth. Round-3's justification invoked an aggregate "8 peers × 9 blocks/min = 72/min" rate. That model is wrong: `block_count` is **per-peer-reader-loop local state** (`src/network/sync.rs:3543`), not a shared counter. Each peer gets its own 12/min budget. A single honest peer on the happy path rarely sends more than 12 BlockResponses to us in 60 seconds outside of pre-confirmation orphan chase or IBD — both of which Change A and the existing `is_ibd_peer` exemption already cover.

Without a correct per-peer rationale, bumping the cap is a change without a justified motivation. Dropped for v1.7.1. If post-ship operation reveals a specific per-peer honest-path scenario that trips 12/min, we can revisit in a point release.

**MAX_BLOCKS_PER_MIN stays at 12.**

## What this does NOT claim

- This is not the last bug in cold-bootstrap. The orphan-chase cascade may still produce behaviors worth tuning (e.g., walk depth, buffer sizes) after Change A un-strikes the seeds.
- This is not a formal proof that no other rate-limit path could fire on a fresh node. We targeted the specific observed failure; if the empirical gate surfaces a different strike source, we iterate.
- Change A does not help **in-process** post-confirmation reorgs — once `ever_confirmed_peer` is `true` for the process lifetime, a BlockResponse cascade from a non-IBD peer during a reorg is still subject to the 12/min cap. (A process restart resets the flag and therefore grants the exemption again on the next boot, which is the intended "long-offline / post-restart" path — see state table row 8. We accept that in-process reorgs past the 12/min cap will strike whoever sends the cascade; no empirical evidence yet that this is a problem, revisit if it becomes one.)

---

## Non-goals

- No re-examination of Fix 2's design or code. v1.7.0's forward-chain redesign remains correct; it was blocked by a transport-layer bug, not by its own logic.
- No change to the strike/ban policy or thresholds. Strikes and bans continue working as designed; they just stop firing on honest pre-confirmation bootstrap orphan-chase traffic.
- No rate-limit constant changes. `MAX_BLOCKS_PER_MIN` stays at 12. Change A is a new `Node` field + a new predicate in the BlockResponse handler; nothing else touched.
