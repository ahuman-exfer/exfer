# v1.5.2 — Cold-bootstrap regression hotfix

**Release type:** Hotfix. Single fix, surgical, targeted.
**Scope:** One fix — prevent the TipResponse handler's legacy `GetHeaders` from racing with an in-flight forward-chain validation on the same (peer, session).
**Upgrade urgency:** Mandatory for anyone doing a cold bootstrap. Existing v1.5.1 nodes that already have chain state are unaffected and can upgrade at leisure.

---

## The regression

v1.5.0 (and the identical-in-behavior v1.5.1) ships a subscriber-routing mechanism introduced for Fix 2 forward-chain validation. The routing decision in `Message::Headers` handling at `src/network/sync.rs:3671` is **session-scoped, not request-scoped**: any `Message::Headers` response from a peer whose (peer_id, session_id) has a subscriber registered is routed to that subscriber, regardless of which of our outbound `GetHeaders` the response actually corresponds to.

The TipResponse handler at `src/network/sync.rs:5224` unconditionally issues a legacy `GetHeaders(start=peer_tip, count=1)` for every TipResponse event received. The sync manager polls every connected peer for their tip every 60 seconds at `src/network/sync.rs:5167`. So during an in-flight forward-chain validation for a peer, a 60-second cadence of legacy `GetHeaders` + legacy response keeps firing, and:

1. The legacy response gets **siphoned away** from main mpsc and routed into the validation task's subscriber queue (because a subscriber exists for (peer, session)).
2. The validation task was waiting for its own response to its own `GetHeaders(start=next_height_in_forward_loop, count=64)`. Instead it pulls the legacy response — a 1-header batch at peer_tip.
3. Comparison at `src/network/sync.rs:5894-5902`:
   - `batch[0].height (peer_tip) != next_height (e.g., 130385)`
   - Returns `TipValidationError::DeliveredInvalidHeader("malformed batch: ...")`
   - `should_strike(DeliveredInvalidHeader) == true` → **strike recorded on honest peer**.
4. Meanwhile the TipResponse handler's own `rx.recv()` for that legacy request times out (10 seconds) because its response was diverted → falls through to the "failed PoW verification" branch at `src/network/sync.rs:5352` → **second strike on the same honest peer**.

### Consequence for cold bootstrap

- `IP_BAN_STRIKE_THRESHOLD = 10` (`src/network/sync.rs:555`).
- Each 60-second cycle produces 1-2 strikes per peer under active validation.
- After ~5-10 minutes, peer reaches 10 strikes and gets banned.
- Forward-chain validation on a fresh node takes 1-7 hours (165K above-checkpoint Argon2 evals).
- **The peer we're trying to validate gets banned by us long before the validation completes.**
- Validation aborts on peer disconnect. Node selects next peer. Same cycle.
- After N × 10 min, seed peer list is exhausted of non-banned peers. Cold bootstrap blocked.

### Not typically bootstrap-blocking

- Upgrade-in-place on existing chain state: node replays from storage, lands at previously-stored tip (300K+), enters SteadyState regime, rarely runs the forward-chain validation path that triggers the collision loop.
- SteadyState nodes CAN still hit the same race whenever a parent-unknown tip-validation runs (only for peers claiming tips above our stored tip that we don't have), but these are rare, and SteadyState validations complete in minutes, so a peer rarely stays in the strike-accumulation window long enough to be banned. Strikes can still accumulate slowly — this hotfix closes that residual surface too.

Cold-install users are the killed-on-contact case. Existing synced users are the occasional-collision case; v1.5.2 is recommended for both, mandatory for the former.

---

## Fix — narrow, surgical

**Prevent the collision at the source.** In the TipResponse handler, consult `tip_validation_coord.active` for the (peer, session) pair **before** issuing the legacy `GetHeaders`. If a validation is already in flight for that pair, skip the entire handler — the in-flight validator owns peer-tip and strike decisions for this peer-session.

### Exact change (spec, not code)

**File:** `src/network/sync.rs`
**Function:** main sync manager loop, `PeerEvent::TipResponse` handler (`sync.rs:5198` onward).
**Location:** immediately after storing the unconfirmed tip, **before** the `node.send_to_session(...GetHeaders...)` call at the current line `sync.rs:5224`.

**Behavior:**

1. Acquire a read on `node.tip_validation_coord.active`.
2. If `(from_identity, session_id)` is present in the active set: do **not** issue the legacy `GetHeaders`, do **not** run the legacy wait loop, do **not** run the fall-through "failed PoW" strike path. Fall out of the handler — the in-flight validation task already owns this peer's tip confirmation.
3. Otherwise, proceed as currently implemented (issue legacy `GetHeaders`, run the 10-second wait loop, dispatch async if dispatch_forward).

This removes the legacy `GetHeaders` that is the race's first ingredient. The subscriber-routing semantics are unchanged; the routing simply no longer has a response to route, because we never emitted the colliding request.

### Invariants preserved

- `should_strike` policy unchanged — still narrow (DeliveredInvalidHeader only).
- Subscriber routing unchanged.
- `run_tip_forward_validation` unchanged.
- `try_reserve`/`release_reservation` unchanged.
- Validation state machine unchanged.
- No change to the wire protocol.
- No change to consensus rules.
- No change to rate-cap regimes or concurrency caps.

### Why this is sufficient

The only ways a Headers response from (peer, session) can reach the subscriber are:
1. The validation task issued a `GetHeaders` itself → expected, safe, this is the intended behavior.
2. The TipResponse handler's legacy `GetHeaders` → eliminated by this fix.
3. Some other code path issuing `GetHeaders` to the same peer while a validation is in flight. Other `GetHeaders` call sites DO exist — `check_shared_block_via_events` at `sync.rs:3964` (common-ancestor binary search) and `run_ibd` at `sync.rs:4392` (IBD header fetch). **Neither currently collides with an in-flight forward-chain validation on the same (peer, session):**
   - `check_shared_block_via_events` IS the common-ancestor search within `run_tip_forward_validation` itself — it shares the same subscriber as the caller, so responses route correctly.
   - `run_ibd` only runs after a peer is confirmed (validation complete, subscriber removed), so no overlap.
   If any future code adds a new `GetHeaders` path touching peers with active validations, it must be audited for the same collision.

With (2) removed, only (1) remains as a first-party writer to the subscriber queue, and the queue becomes a single-writer pipeline. No race.

## Scope bounds (important)

**Do NOT fix as part of this hotfix:**

1. **Request-scoped subscriber routing.** The *real* fix is to make the subscriber routing decision per-request rather than per-session — e.g., tag each outgoing `GetHeaders` with a nonce or expected start_height, and route responses to the matching in-flight request. This is a larger change that touches the wire-level routing layer. **Defer to v1.6.0** where it can be designed alongside any other protocol-level cleanup.

2. **Narrower `should_strike` on `DeliveredInvalidHeader`.** A "malformed batch" during a subscriber race is not actually a peer serving invalid content — it's our own routing confusion. One could argue `should_strike` should sub-classify `DeliveredInvalidHeader` so that race-induced failures don't strike honest peers. With the primary fix (this hotfix), the race is eliminated and this distinction no longer matters; striking on genuinely-delivered-invalid headers remains correct. **Note for future but do not implement here** — the primary fix eliminates the need.

3. **NoSlotAvailable retry path.** Currently NoSlotAvailable leaves the peer with `confirmed=false` and no auto-retry. Eventually the 60s poll will re-trigger the handler; with this hotfix the handler will defer to the in-flight validation, and once that completes and releases the slot, the next poll will succeed. This behavior is acceptable — do not add retry queues or explicit re-scheduling.

## Tests required

### Unit test

Since the TipResponse handler is inline in the main sync manager loop (`sync.rs:5199` onward) and `send_to_session` / `record_ip_strike` are both real side-effects that are awkward to mock from a unit test, **extract the guard into a tiny helper** as part of this hotfix:

```rust
// In src/network/sync.rs or a small helper module.
impl TipValidationCoordinator {
    pub async fn is_active(&self, peer: PeerId, session_id: u64) -> bool {
        self.active.lock().await.contains(&(peer, session_id))
    }
}
```

The TipResponse handler uses `coord.is_active(from_identity, session_id).await` as the guard. `is_active` is then the target of direct unit testing:

**`is_active_returns_true_after_reserve_and_false_after_release`**
- `coord.try_reserve(peer, sid).await` → assert `coord.is_active(peer, sid).await == true`
- `coord.release_reservation(peer, sid).await` → assert `coord.is_active(peer, sid).await == false`
- Fresh coord never-reserved: assert `coord.is_active(peer, sid).await == false`

This is trivial to test and exercises exactly the guard condition.

For the integration-level unit test of "handler skips GetHeaders when active":

**`tipresponse_handler_skips_getheaders_when_validation_active`** (integration test in `tests/tipresponse_race_integration.rs`)

Set up:
- A full `Node` bound to `127.0.0.1:<port>` on an ephemeral port.
- A mock peer that connects and completes handshake, then sends `TipResponse` for a height above the node's tip.
- Before injecting the TipResponse, reserve `(peer, session)` via `node.tip_validation_coord.try_reserve`.

Assert:
- Observe the mock peer's inbound message stream — no `Message::GetHeaders` arrives for this peer during the handler execution window.
- `node.record_ip_strike` is not called for this peer (inspect `node.ip_abuse` state post-injection).
- Peer's `tip.confirmed` remains `false` (unchanged behavior).

### Integration test

**`tipresponse_race_does_not_strike_honest_peer`** (new test in `tests/tipresponse_race_integration.rs`)

Set up:
- A `PeerRegistry` with one peer session.
- Mark a validation as active for that peer via `tip_validation_coord.try_reserve`.
- Dispatch two `PeerEvent::TipResponse` events for that peer, simulating two consecutive 60-second polls.

Assert:
- After both events, the peer's `record_ip_strike` count is 0.
- No `DeliveredInvalidHeader: malformed batch` error logged.
- `tip_validation_coord.active` still contains the (peer, session) entry (untouched by the TipResponse handler).

### Regression test — cold bootstrap

**`v1_5_2_cold_bootstrap_reaches_tip_without_strike_storm`** (new test in `tests/cold_bootstrap_regression.rs` or a manual test procedure if integration harness is too heavy)

**A. Full integration (preferred if feasible):**

### Strike regression sanity

**`honest_peer_no_strike_on_60s_poll_during_validation`** (new test, or a focused variant of the integration test above)

Construct the exact timing sequence:
- t=0: validation task installed a subscriber for (peer, session).
- t=1: `PeerEvent::TipResponse` for (peer, session) arrives at sync manager.
- Assert: handler does not call `send_to_session(Message::GetHeaders)` for this peer.
- Assert: handler does not invoke the 10-second `rx.recv` wait loop.
- Assert: handler does not fall through to `record_ip_strike`.

>
> If you are currently running v1.5.1 and your node is already synced to the network, v1.5.1 is not typically bootstrap-blocking for you — the regression mainly affects fresh installs, not nodes that already have chain state. >

A regression in v1.5.0's Fix 2 (tip-hardening) code path caused fresh nodes to mistakenly strike the honest peers they were trying to sync from, eventually banning all seed peers and blocking cold bootstrap. v1.5.2 fixes this by skipping the colliding request that triggered the race.
>

Fast-path review. The expert already knows the Fix 2 architecture. Key things to review:

1. **The fix location is correct** — the guard belongs immediately before the `send_to_session(...GetHeaders...)` at `sync.rs:5224`, not anywhere later in the handler.
