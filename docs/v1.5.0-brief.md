# v1.5.0 — Network topology + consensus hardening

**Release type:** Feature + hardening. Ship after v1.4.2 bake.
**Scope:** Two fixes — one headline protocol change (random inbound eviction) and one consensus-adjacent hardening (sync.rs unknown-parent tip).
**Upgrade urgency:** Recommended for all node operators. Not a security patch — no known in-the-wild theft or DoS exploit. Fixes a structural centralization vector and a class of consensus edge cases.

---

## Fix 1 — Random inbound peer eviction (HIGH priority, headline change)

### The problem

Empirical finding from the v1.4.2 post-release analysis (2026-04-18): the network's peer graph is colonized by a single operator.

- Of 1,048 "healthy" (crawler-visible) nodes, **~1,027 (98%) are on Alibaba Cloud International (AS45102)** — spread across Virginia, Frankfurt, and Manila DCs but all under one ASN, almost certainly one operator's fleet running mass-provisioned instances.
- The Alibaba fleet aggressively dials every address it discovers via `Addr`-gossip peer-exchange.
- Of our seed nodes' 256 inbound slots, **~90% (~235 of 260) are held by Alibaba IPs**.
- Cross-testing confirmed the same fleet has filled inbound slots on most other Exfer nodes network-wide: when we probe random non-healthy IPs, TCP behaviour (86% timeout / 14% RST) matches what we'd see from nodes whose accept queue is saturated.
- Result: **new non-Alibaba miners trying to peer with any public node are rejected** with `"Max inbound peers reached"`. They try a few times, give up, and either run peer-isolated (still mining, but with degraded block propagation and high orphan rate) or leave the network entirely.

The colonization is a consequence of protocol design, not operator malice: `MAX_INBOUND_PEERS=256` slots allocated first-come-first-served, with `MAX_INBOUND_PER_IP=1` enforced per IP but not per fleet. Any operator with more than ~256 distinct IPs making aggressive outbound connections permanently holds all slots on every node they reach.

### Why this matters

- **De facto topology centralization.** The largest operator controls ingress to every public node.
- **Reinforces observed hashrate concentration.** Non-Alibaba miners orphan more because their blocks propagate through Alibaba relays rather than direct peer links.
- **Makes the seed-node layer a single point of failure.** If the Alibaba operator chose, they could partition the network.
- **Violates permissionlessness in practice if not in rule.** New participants can't join the peer graph on equal terms.

### Non-goals

- **Do not discriminate by ASN, geography, or any operator attribute.** A network that rejects peers based on who they are is doing censorship under a decentralization costume. Any ASN-cap / geo-diversity / diversity-scoring approach is out of scope and should not be proposed again.
- **Do not require new centralized infrastructure.** No local seed DNS, no centrally-maintained peer reputation lists, no trusted "good peers" allowlists.
- **Do not change outbound peer selection logic.** We still dial whoever we want to dial on the outbound side. Scope is inbound slots only.

### The fix: random eviction on full inbound

When a new inbound peer completes handshake and our node is at `MAX_INBOUND_PEERS`, randomly evict an existing eligible inbound peer and admit the new one. At steady state this means each operator's share of slots approaches their share of connection attempts, not 100% of whoever-got-in-first. Colonization breaks in expectation over `(MAX_INBOUND_PEERS × ln(2)) / new_arrival_rate` seconds — typically minutes to hours depending on network traffic.

This is the same mechanism Bitcoin Core uses (`AttemptToEvictConnection` in `net_processing.cpp`), though Core uses utility-based ranking rather than pure random. We use pure random for v1.5.0 because:
- Simpler to implement (<200 lines)
- Simpler to reason about (no fingerprinting edge cases)
- No preferences → no censorship surface
- Utility-based can be revisited in a later release if random isn't effective

### Exact eviction trigger conditions

Eviction fires if **ALL** of the following hold at the point where `attach_session` would be called in `handle_inbound` (see `src/network/sync.rs:2711`):

1. The inbound handshake completed successfully (valid `HelloMsg`, matching `genesis_block_id`, passing signature verification, matching `PROTOCOL_VERSION`).
2. The peer's identity is not banned (existing `is_identity_banned` check still applies, BEFORE eviction consideration).
3. The peer's source IP is not banned (existing `is_ip_banned` check still applies, BEFORE eviction consideration).
4. `peers.inbound_count() >= MAX_INBOUND_PEERS`.
5. `peers.inbound_count_for_ip(addr.ip()) < MAX_INBOUND_PER_IP` (the per-IP cap is independent and still enforced — an IP already at its limit cannot force eviction to add another connection from the same IP).
6. **Duplicate-identity gate (new).** The peer's `identity` is not already attached as a `LogicalPeer` with an active session. If it is, eviction is **skipped entirely** — fall through to the existing `attach_session` path which returns `SessionAttachResult::ReplacedExistingSession` and signals the old session's shutdown flag. That path frees the old session's slot naturally, and the new session takes it. No unrelated third-party peer gets evicted for a same-identity reconnect.

If any condition fails, the existing reject / ban / duplicate-session / replace behaviour applies — **do not attempt eviction to bypass other gates**.

**Ordering note (critical):** the duplicate-identity check (condition 6) must run under the same `peers` lock as the eviction decision. A reconnect from an already-connected identity arriving at exactly the moment slots fill must not cause an unrelated peer to be evicted. Path: (a) acquire peers lock, (b) check identity-duplicate first, (c) if duplicate → release lock, proceed to `attach_session` (which handles the replace), no eviction, (d) if not duplicate and slots full → check all other gates, then evict, then attach.

### Pre-handshake TCP accept

Current behaviour (`src/network/sync.rs:2612-2629`): reject TCP accept when `inbound_count + pending_inbound_sockets >= MAX_INBOUND_PEERS`.

New behaviour: allow an overcommit headroom for in-progress handshakes. Introduce:

```rust
pub const EVICTION_PENDING_HEADROOM: usize = 32;
```

Change the accept-time check to reject only when `inbound_count + pending_inbound_sockets >= MAX_INBOUND_PEERS + EVICTION_PENDING_HEADROOM`. Handshakes that fail or time out (at `HANDSHAKE_TIMEOUT_SECS=5`) release their pending slot and never reach the eviction logic. Handshakes that succeed reach `attach_session` and trigger eviction only if we're still over the core cap.

**Rationale for `EVICTION_PENDING_HEADROOM = 32`.** Handshake timeout is 5 seconds, so at most 32 pending slots can exist concurrently before the oldest times out and frees space. 32 ≈ 12% of `MAX_INBOUND_PEERS`, which absorbs a typical burst (dozens of peers reconnecting simultaneously after a brief network hiccup) without letting an adversary pin arbitrary resources. Smaller values (e.g., 8) would reject legitimate handshakes during modest bursts; larger (e.g., 128) would let a handshake-flood attacker hold more partially-established sockets and more HMAC-verification work in flight. 32 is the smallest power-of-two-ish value that comfortably covers observed natural burst sizes with zero measurable cost to legitimate operators.

This bounds worst-case memory and CPU impact of eviction-pending: at most 32 concurrent mid-handshake peers above cap, each bounded by the 5-second handshake timeout. Memory per pending is small (<64 KiB — framed HelloMsg buffers + socket state).

### Random selection mechanism

Candidate pool for eviction: all established inbound peers satisfying **ALL** of:

1. **Not in handshake.** Only fully-attached peers (`is_outbound == false` in `PeerSession`, session established).
2. **Not recently admitted.** `Instant::now() - session.established_at >= EVICTION_MIN_AGE_SECS`. Proposed value: `60` seconds. This prevents thrash where a burst of handshakes evicts each other in a loop.
3. **Not the active IBD peer.** If `node.active_ibd_peer` is `Some((identity, session_id))` and the candidate matches, skip — evicting mid-IBD would waste work and potentially stall chain sync.

Selection: uniform random over the candidate pool via `rand::thread_rng().gen_range(0..candidates.len())`. Do NOT use any weighting, reputation, age-ordered selection, or similar — pure uniform random.

If the candidate pool is **empty** (all inbound peers are within `EVICTION_MIN_AGE` or are IBD — an edge case but possible during a traffic burst), fall back to existing reject behaviour: log `"Max inbound peers reached, no eligible eviction candidates, rejecting X"` and drop the new peer. Do not evict a protected peer.

### Eviction execution

Under a single lock on `peers`, atomic sequence:

1. Re-check `inbound_count() >= MAX_INBOUND_PEERS` (guard against race with another concurrent handshake completion).
2. Enumerate eligible candidates per the rules above.
3. If empty → release the new peer's pending slot, return `"Max inbound peers reached"` error, log at `INFO` level.
4. Random-index into candidates, get `victim: PeerSession`.
5. Set `victim.shutdown.store(true, Ordering::Release)`. The victim's reader/writer/supervisor tasks observe the flag at their next check and unwind. **Timing is eventual, not prompt.** The supervisor polls the flag on a 1-second tick, but the reader may be blocked inside a frame read (up to `FRAME_READ_TIMEOUT_SECS=10`) or chunk wait (up to `CHUNK_READ_TIMEOUT_SECS=5`) before returning from its current syscall and observing the flag on the next loop iteration. Worst-case clean unwind is bounded by the current read-deadline stack (order ~10-20 seconds), not by the supervisor tick. This is acceptable: the slot is logically reclaimed immediately (step 6 — new peer admitted, logical-peer registry updated), and the stale reader's resources (socket, frame reservation) drain within existing transport deadlines.
6. Remove the victim's session from the `LogicalPeer` registry — call existing `detach_session(identity, session_id)` or equivalent. This removes the logical-peer entry but does NOT synchronously reap the reader/writer tasks; they exit on their own after observing the shutdown flag.
7. Call `attach_session` for the new peer as normal.
8. Log at `INFO` level: `"Evicted inbound peer {victim_addr} (identity {victim_identity[..8]}, age {age_secs}s) to admit {new_addr}"`.

No RPC or notification is sent to the evicted peer before shutdown. They observe connection close and can reconnect if they want; at worst they experience one unexpected disconnect.

**Memory ordering note.** Step 5 specifies `Ordering::Release` on the store. The shutdown flag has no happens-before dependency on other memory — the reader's only action on observing `shutdown=true` is to exit its loop, which doesn't require seeing any other prior stores to be correct. Existing readers (`src/network/peer.rs`) use `Ordering::Relaxed` on the load, and this is sufficient. `Release` on the writer is specified as defensive style: it pairs naturally with any future Acquire loader that may need happens-before, and has zero cost on x86 and trivial cost on ARM. `Ordering::Relaxed` would also be correct given the current reader pattern — both are acceptable; pick `Release` for forward-compatibility.

**Frame budget release on eviction (critical).** The evicted peer's in-flight `FrameReservation` (v1.4.2 Fix 3, `src/network/frame_budget.rs`) is released via its `Drop` impl when the reader task exits during shutdown. **Eviction must not bypass this path.** Specifically:

- Do NOT manually detach the `PeerSharedState`, drop the `PeerBudget` reference, or free socket resources from the eviction path.
- The correct sequence is: set the shutdown flag (step 5), update the peer registry (step 6), and return. The reader task, on its next iteration, observes `shutdown=true` and returns from `read_message_after_type`, which drops the `FrameReservation` naturally. The per-peer and global in-flight byte counters are decremented only via this Drop.
- Bypassing the reader's unwind (e.g., by forcibly cancelling the task or reaping shared state early) would leak reservation bytes against the global 128 MiB / per-peer 16 MiB budgets, eventually starving legitimate frames.
- The peer-supervisor structure (`run_peer_supervisor` in `sync.rs:3469`) already coordinates clean shutdown between reader/writer/supervisor via the shared `shutdown` AtomicBool. Eviction reuses this existing mechanism unchanged.

### Mid-handshake peer handling

- **Pending handshakes are not eviction candidates.** A pending handshake hasn't been authenticated yet — evicting it to admit another pending handshake buys us nothing.
- **Pending handshakes do count against the accept-time overcommit limit** (`MAX_INBOUND_PEERS + EVICTION_PENDING_HEADROOM`).
- **If handshake fails** (timeout, signature mismatch, duplicate identity, banned identity), the pending socket is released as it is today. No eviction consideration.

### Interaction with rate limits and bans

All existing rate-limit / ban gates apply **before** eviction considers the new peer:

- **`is_ip_banned(addr.ip())`** → reject at TCP accept time (already enforced at `sync.rs:2606`). No change.
- **`is_identity_banned(peer.identity)`** → reject at handshake completion (already enforced at `sync.rs:2677`). No change. This specifically prevents an attacker from forcing eviction of good peers by spamming handshakes from rotating-identity bots under banned identities.
- **`MAX_INBOUND_PER_IP=1`** → the peer's IP is checked independently; a new peer whose IP already has a slot cannot force eviction.
- **Peer-layer rate limits** (the existing `"Authenticated peer X disconnected (rate limit exceeded) — recording IP strike"` path): triggered after attachment, unchanged. An evicted peer that reconnects and behaves badly still accumulates strikes.

**Important interaction:** the existing strike logic (`record_ip_strike` at `sync.rs:795`, keyed by IP with identity-ban as a side effect) should NOT treat "evicted then reconnected" as a strike. The old session's shutdown is an administrative action by us, not misbehaviour by the peer. If any existing strike-recording path fires on clean disconnects, it must be adjusted to exclude eviction-induced disconnects.

### Scope bound — explicit (read twice)

This change applies ONLY to the inbound slot-accept path. It does NOT modify:

- **Outbound peer selection.** We choose who to dial, how often, in what order. That logic in the outbound manager is unchanged. Outbound peer selection draws targets from `Addr`-gossip independent of current inbound session state; inbound churn (eviction or otherwise) does not affect outbound target selection.
- **`MAX_OUTBOUND_PEERS=8`.** Unchanged.
- **Peer-exchange (`Addr` messages).** The gossip protocol that teaches peers about each other is unchanged.
- **Handshake protocol.** Same 3-step handshake with the same timeouts.
- **Frame budget / transport layer.** The v1.4.2 Fix 3 in-flight-buffer budget applies unchanged.
- **Bans, abuse tracking, rate limits.** Existing logic runs unchanged; eviction is gated behind all of them.

If an edge case suggests touching any of the above, it's out of scope and needs a separate proposal.

### Error semantics

- **Eviction succeeds:** `INFO` log line with both sides of the exchange (victim's addr + identity prefix + session age, new peer's addr).
- **Eviction falls back to reject (no eligible victims):** `INFO` log line noting this, reject with the existing `"Max inbound peers reached"` message.
- **No new error codes or new user-facing behaviour for the evicted peer** — they just see a clean TCP close. The event is only visible in operator logs.

#### Unit tests (in `src/network/sync.rs` or a new `src/network/eviction.rs` module)

8. **`eviction_respects_max_inbound_per_ip`** — inbound_count_for_ip(X) already at cap; new peer from X. Assert reject without eviction.
9. **`eviction_races_atomically`** — simulate two concurrent handshakes completing at the same instant when one slot is free. Assert exactly one evicts and admits, the other hits the re-check and rejects cleanly (no double-admission, no deadlock).

#### Integration test (new file, `tests/eviction_integration.rs`)

10. **`colonization_resistance`** — spin up an in-memory `Node` with `MAX_INBOUND_PEERS=16` and `EVICTION_MIN_AGE=1s` for speed. Inject 16 simulated inbound handshakes from `fleet_A` identity prefix, filling all slots. Then inject 100 handshakes over 10 seconds split 50/50 between `fleet_A` and `fleet_B`. After the burst, assert `fleet_B` holds at least 30% of slots (within 3σ of the expected 50%) — confirms colonization broke.
11. **`no_thrash_on_burst`** — inject 200 simultaneous handshakes under load. Assert no peer is evicted within `EVICTION_MIN_AGE` of being admitted.
12. **`evicted_peer_clean_shutdown`** — evicted peer's reader/writer tasks must exit cleanly within the existing transport deadlines (bounded by `FRAME_READ_TIMEOUT_SECS + CHUNK_READ_TIMEOUT_SECS + supervisor_tick`, i.e., ~20s worst case, typically sub-second when the reader is idle between frames); no orphan tasks, no FD leaks; the associated `FrameReservation` is released via its `Drop` impl (global and per-peer in-flight counters return to pre-eviction values). Test threshold: **20 seconds** (not 2). The assertion is "eventual clean release bounded by current read deadlines" — tightening this would require reducing the transport timeouts, which is out of scope for v1.5.0.
13. **`long_horizon_colonization_resistance`** — `MAX_INBOUND_PEERS=16`, `EVICTION_MIN_AGE=1s` for test speed. Simulate `fleet_A` aggressively redialing at 20 handshakes/sec continuously for 60+ seconds, and `fleet_B` arriving at a slower 2 handshakes/sec (10:1 ratio). **Methodology (revised per review):** a single 60-second window has high variance — with 16 slots and 2/(20+2)=9% arrival share, expected fleet_B slots ≈ 1.45, with integer-quantization making "strictly > 0" a weak claim. Use instead **time-averaged slot occupancy over the trial window** (sampled once per second, averaged across all 60 samples, so the sample size is 60 independent observations) AND **repeat the full trial 10 times with distinct RNG seeds**. Assert the mean-of-means (600 samples total) is within ±30% of the arrival-rate share (~9%), i.e., lies in [0.063, 0.117] slot-fraction. Also assert the minimum per-trial time-averaged share is > 0.01 (confirming fleet_B is never fully shut out). This is the core property test proving eviction breaks permanent colonization under sustained adversarial dialing; the old single-trial formulation was not statistically reliable.

### Rollout considerations

- **No protocol change.** Existing peers on v1.4.2 or earlier interoperate identically. Eviction is purely node-local behaviour — a peer disconnected by us just sees a TCP close, same as any other reason.
- **No flag / feature gate needed.** The change is safe and beneficial immediately. Enable by default.
- **Monitoring:** emit a log-level metric for eviction rate (count of evictions per minute, per hour). Operator should see a few evictions per minute under normal traffic, spiking during colonization pressure.
- **Expert re-review:** network-topology work, less consensus-adjacent than Fix 2, but still worth an expert pass on the locking / atomic path (step 1 re-check under the peer lock is the race-sensitive bit).

## Fix 2 — `sync.rs` unknown-parent tip hardening (consensus-adjacent, route to expert)

**Status: draft for expert review BEFORE implementation.** This touches the IBD decision path and consensus-adjacent code. Do not ship until the external reviewer has signed off on both the threat model and the proposed mitigation. Implementation should follow the same rhythm as v1.4.2: expert-approved spec → implementation → unit + integration tests → ≥1 hour fuzz pass → expert re-review on the diff.

### The problem

Current tip-confirmation flow (`sync.rs:4906-5034`):

1. Peer sends `TipResponse { height, block_id, cumulative_work }`. Stored as `confirmed: false`.
2. Sync manager sends `GetHeaders { start_height: height, max_count: 1 }` to fetch a single header at the claimed height.
3. On `HeadersResponse`:
   - Check `header.block_id == claimed.block_id` and `header.height == claimed.height` — binds the peer to a specific header.
   - Difficulty validation splits on whether we have the parent:
     - **Parent known** (our chain reaches at least to `header.prev_block_id`): verify `header.difficulty_target == cached_expected_difficulty(parent, height)`. Strict.
     - **Parent unknown** (peer claims a height we haven't reached): **accept any difficulty target** the peer claims. Only the PoW is verified against that claimed target (`sync.rs:4964-4969`).
4. If PoW verifies: mark `confirmed: true`. Override peer's claimed `cumulative_work` with either our stored work for the block or `our_tip.cumulative_work + work_from_target(header.difficulty_target)`.
5. With `confirmed: true`, the sync manager considers this peer a valid IBD target and may trigger IBD from them.

### Attack surface

A malicious peer can:

1. Compute one Argon2id hash with an artificially easy `difficulty_target` (~100ms compute cost — trivial).
2. Produce a well-formed block header at any arbitrary claimed height `H_malicious`, with:
   - `difficulty_target = 0xFFFF_FFFF_FFFF_FFFF_...` (or any easy target that still admits a low-nonce solution)
   - Valid PoW for that easy target (trivially found)
   - Any `prev_block_id` (we don't verify it)
3. Send `TipResponse { height: H_malicious, block_id, cumulative_work: arbitrary_high }`.
4. When we `GetHeaders` for the single header at `H_malicious`, deliver the easy-PoW header.
5. Our validation accepts it (parent unknown → any difficulty allowed, PoW verifies for that target).
6. We mark the tip `confirmed: true`.

Impact:

- Forces us into IBD with the attacker. We set `active_ibd_peer = (attacker_id, session)`. While this is set, we refuse to start IBD from other peers.
- The attacker can't actually deliver a real chain up to `H_malicious` — their chain doesn't exist. IBD will fail, but only after some timeout and bandwidth consumption.
- **Sustained harm vector:** an attacker controlling many identities can chain these attacks — as soon as one IBD times out and the `active_ibd_peer` lock releases, another identity lands their fake tip and we're back in bogus IBD. Worst-case sustained IBD lock-in prevents us from syncing from legitimate peers for as long as the attacker can cycle identities through the `identity_bans` cooldown.
- The existing defense of clamping the override `cumulative_work` to `our_work + work_from_target(easy_target)` helps — the attacker can't claim arbitrarily high work. But since `work_from_target` on an easy target is a tiny value, the result is `our_work + epsilon`, which still beats `our_work` by `epsilon`, which still triggers `is_better_chain` and starts IBD. **The clamp bounds the claim but does not prevent the IBD trigger.**

### Why the current mitigation is insufficient

The code comment at `sync.rs:4966-4968` acknowledges the gap:

> *"Parent unknown (peer is ahead). We can't verify the exact difficulty, but PoW validity still proves real work was done for the claimed target. The full chain will be validated during IBD."*

This reasoning is wrong because:

- **"Real work" on an easy target is negligible work.** ~100ms of Argon2id per attempt is not a meaningful cost relative to the damage of a forced IBD cycle (minutes to hours of our bandwidth + CPU).
- **"Full chain validated during IBD"** is true but comes AFTER we've committed to that peer. The commitment itself is the exploitable primitive.
- **The single-header validation gives no evidence the peer actually has the chain.** A peer that holds one easy header has demonstrated nothing about the 100,000 real blocks they'd need to also deliver.

### Required fix: common-ancestor + exact-forward header validation (revised per expert review)

The fix structure was revised substantially in expert review. Rather than validating a backward-walking chain with heuristic caps (the earlier draft's `K`-cap and `0.5` factor approach, now dropped), we use **exact consensus difficulty** on a forward walk from a common ancestor. This is both simpler and stronger: there is no heuristic margin for an attacker to squeeze through.

**When peer's claimed tip has unknown parent** (our chain doesn't reach `header.prev_block_id`):

1. Do NOT mark `confirmed: true` based on one header alone.
2. **Locate anchor.** Two sub-paths depending on our local state:

   **2a. Running-node path (our local tip ≥ ASSUME_VALID_HEIGHT, or `--verify-all`):** Reuse the existing height-based binary-search mechanism `find_common_ancestor_via_events` (`src/network/sync.rs:3707`). It probes specific heights via the existing `GetHeaders` wire type (`start_height + max_count`) and asks the peer to return the header at that height; if the peer's block_id at that height matches ours, we recurse higher; otherwise lower. Converges in `O(log Δheight)` probes against the current wire protocol. Call the returned height `anchor_height` and our canonical block at that height `anchor_header`. (Note: our `GetHeadersMsg` is `{ start_height: u64, max_count: u32 }`; the Bitcoin-style locator + `stop_hash` flow is explicitly NOT proposed here, which keeps v1.5.0 wire-compatible with v1.4.2.)

   **2b. Cold-bootstrap path (our local tip < ASSUME_VALID_HEIGHT, assume_valid enabled):** The existing binary search only probes heights we have, and at cold bootstrap we have only genesis (or up to our current partial tip, below the checkpoint). Binary search cannot reach the checkpoint. Instead:
   - Send `GetHeaders { start_height: ASSUME_VALID_HEIGHT, max_count: 1 }` to the peer. Expect one header in the response.
   - Verify the returned header's `block_id == ASSUME_VALID_HASH` (the hardcoded checkpoint hash in `src/types/mod.rs:36`). **If it doesn't match, the peer has delivered an invalid header at a well-known consensus-anchor height — record a strike and abort this tip-validation.**
   - Use the fetched header (`anchor_header = fetched_checkpoint_header`, `anchor_height = ASSUME_VALID_HEIGHT`) as the anchor for forward validation. The hardcoded checkpoint hash is our trust anchor; a peer supplying a header hash-matching `ASSUME_VALID_HASH` at the correct height is serving the real checkpoint block's header.
   - Rationale: `src/types/mod.rs:34-43` hardcodes only the block-id, not the full header (version, prev, merkle_root, timestamp, difficulty_target, nonce). Forward-difficulty validation needs `difficulty_target` from the checkpoint header, so we must fetch it from the peer. Block-id authentication + the single exact-byte comparison against `ASSUME_VALID_HASH` makes the fetched header as trustworthy as the hardcoded constant itself — any other header bytes would produce a different SHA256, so the peer cannot substitute.
   - Once the authenticated checkpoint header is in hand, proceed directly to step 3 (forward validation from `anchor_height = ASSUME_VALID_HEIGHT` up to `peer_claimed_tip`).
3. **Validate the forward header chain `(anchor → peer_tip)` against an in-memory overlay.** The existing helpers `cached_expected_difficulty` (`sync.rs:667`) and `expected_difficulty` (`consensus/difficulty.rs:56`) are **storage-backed** — they call `storage.get_header(prev_block_id)` and (at retarget boundaries) walk `RETARGET_WINDOW - 1 = 4319` ancestors through `ChainStorage`. The forward headers we are validating are NOT in storage yet, so these helpers cannot be used unchanged.

   **Required support work (part of Fix 2):** introduce a temporary in-memory `ForwardHeaderOverlay` used only during tip-validation. Shape:
   ```rust
   struct ForwardHeaderOverlay<'s> {
       storage: &'s ChainStorage,               // fallback for headers at/below anchor
       overlay: HashMap<Hash256, BlockHeader>,  // keyed by block_id; anchor + forward headers validated so far
   }
   impl ForwardHeaderOverlay<'_> {
       fn get_header(&self, id: &Hash256) -> Result<Option<BlockHeader>, StorageErr> {
           if let Some(h) = self.overlay.get(id) { return Ok(Some(h.clone())); }
           self.storage.get_header(id)
       }
   }
   ```
   The overlay is seeded with `anchor_header` at tip-validation start (keyed by `anchor_header.block_id`). Each forward header is inserted into the overlay AFTER its difficulty/PoW check passes (so the overlay only contains validated headers — an invalid header is never a source for subsequent retarget lookups).

   **Add a variant of `expected_difficulty` that takes the overlay:**
   ```rust
   pub fn expected_difficulty_overlay(
       overlay: &ForwardHeaderOverlay,
       prev_block_id: &Hash256,
       height: u64,
   ) -> Result<Hash256, DifficultyError> { /* same logic as difficulty.rs:56 but all get_header calls go through overlay */ }
   ```
   This is a mechanical copy of the existing function that swaps `storage.get_header` for `overlay.get_header`. The retarget math is unchanged. (If preferred, refactor `expected_difficulty` to take a trait `HeaderSource` with impls for both `ChainStorage` and `ForwardHeaderOverlay` to avoid duplication — reviewer's choice; I'd accept either. The spec requires the capability, not the specific factoring.)

   **Retarget-window lookback below the anchor.** A retarget boundary at forward height `H` needs `RETARGET_WINDOW - 1 = 4319` ancestor timestamps back from `H-1`. If `anchor_height = H - k` where `k < 4319`, the lookback chain walks: (a) through `overlay` for the first `k` headers above anchor, (b) through `storage` for the remaining `4319 - k` headers below anchor. The overlay's `get_header` fallback to `storage` handles this automatically, provided the anchor's prev-chain is in storage (which is always the case when the anchor is at or below our local tip — the running-node path 2a — or when the anchor is the checkpoint and our local chain has reached the checkpoint via IBD, as part of the cold-bootstrap path 2b's normal completion). For cold-bootstrap where our local storage is still below anchor = `ASSUME_VALID_HEIGHT`, this is the one constraint: the retarget calculation for forward headers in the first retarget window above the checkpoint might need pre-checkpoint headers we don't have. Mitigation: cold-bootstrap subpath 2b is defined as "fetch the checkpoint header," and at the checkpoint height (130_000), the next retarget boundary is at `ceil(130_001 / 4320) × 4320 = 132_480`, i.e., 2,480 blocks of straight-copy-parent-target before any retarget math is needed. Once IBD has completed even the first 2,480 blocks, the rest of the retarget window is present in storage. For validations whose forward chain crosses the first post-checkpoint retarget boundary during the initial bootstrap, we extend the overlay downward: fetch `RETARGET_WINDOW - 1 = 4319` pre-checkpoint headers from the peer (one extra GetHeaders round-trip for a 68-batch fetch).

**Strict hash-chain authentication of pre-checkpoint headers (critical):** these headers are NOT Argon2-verified because they are below `ASSUME_VALID_HEIGHT` and covered by the checkpoint trust. They MUST be authenticated by strict hash-chain linkage from the already-authenticated checkpoint header:

1. Start from `anchor_header` (the checkpoint header whose block_id has been verified equal to `ASSUME_VALID_HASH`).
2. Expect the peer's delivered batch to contain headers at heights `[ASSUME_VALID_HEIGHT - 4319 .. ASSUME_VALID_HEIGHT - 1]` in order.
3. Walk the batch **from newest (closest to anchor) to oldest**. For each header `h`:
   - Verify `h.block_id` (computed by hashing `h`'s fields) equals the `prev_block_id` of the next-newer header already authenticated (starting from `anchor_header.prev_block_id` for the first step).
   - If the hash match fails, **record a strike and abort the attempt**: the peer has delivered a header that does not chain back to the authenticated checkpoint.
4. Only after ALL 4319 headers have chained back cleanly, insert them into the overlay.

This is a strict block_id == SHA256(header_bytes) == next.prev_block_id check at every step. No tolerance, no gap. The property we rely on: SHA256 pre-image resistance means a peer cannot produce 4319 headers that chain correctly back to `ASSUME_VALID_HASH` without them being the real canonical pre-checkpoint headers. Once chained, their presence makes retarget lookback well-defined. (Expert confirmed this is the expected authentication model in sign-off.)

   **Per forward header `h_i`, the checks are:**
   - **Chain integrity:** `h_i.prev_block_id == h_{i-1}.block_id` (or `anchor_header.block_id` for `h_0`), `h_i.height == h_{i-1}.height + 1`.
   - **Exact difficulty:** `h_i.difficulty_target == expected_difficulty_overlay(&overlay, &h_i.prev_block_id, h_i.height)`. No heuristic multiplier, no tolerance, no cap — exact equality against the consensus function.
   - **PoW validity:** Argon2id `h_i.pow_hash < h_i.difficulty_target`.
   - **On success:** insert `h_i` into `overlay` keyed by `h_i.block_id` before advancing to `h_{i+1}`.

4. **Derive verified cumulative work (new, per review).** Once the entire forward chain `(anchor..=peer_tip)` passes all three checks, compute:
   ```
   verified_cumulative_work = anchor_work + Σ_{i=0..n} work_from_target(h_i.difficulty_target)
   ```
   using the existing `work_from_target` at `consensus/difficulty.rs:206`. `anchor_work` is `storage.get_cumulative_work(anchor_header.block_id)` for path 2a, or the hardcoded `ASSUME_VALID_CUMULATIVE_WORK` that must be added alongside `ASSUME_VALID_HASH` in `types/mod.rs` for path 2b (a new constant — see note below). **Overwrite the peer-supplied `TipResponseMsg.cumulative_work` with `verified_cumulative_work`** before storing on the peer-tip state. From this point forward, any `is_better_chain` / `peer_tip.cumulative_work > our_tip.cumulative_work` comparison uses only the verified value; the peer-supplied value is discarded.

   *Why this matters:* the earlier draft silently allowed peer-supplied `cumulative_work` to survive through confirmation. Even though forward validation structurally prevents a cheap-header attack, carrying the peer-supplied value creates a second ingress point. Tying the stored work to the validated chain eliminates it.

   *Note on `ASSUME_VALID_CUMULATIVE_WORK`:* alongside the existing `ASSUME_VALID_HEIGHT` and `ASSUME_VALID_HASH` in `src/types/mod.rs:36-43`, add a third hardcoded constant recording the cumulative work at the checkpoint height on the canonical chain. Required for path 2b to compute `verified_cumulative_work` without walking storage below anchor. Computed once from a trusted canonical node at release-build time.

   **Runtime verification guard (release-hardening, per expert sign-off):** a bad `ASSUME_VALID_CUMULATIVE_WORK` constant would not break header validation but would quietly skew IBD `is_better_chain` decisions for any path 2b bootstrap. Guard against a mistaken constant with both a build-time check and a runtime assertion:

   - **Runtime assertion.** When the node's local chain reaches `ASSUME_VALID_HEIGHT` through normal block-by-block validation (whether from genesis or post-IBD completion), compute the actual cumulative work stored at that height and assert it equals `ASSUME_VALID_CUMULATIVE_WORK`. Location: the existing `assume_valid_verified` hook — whatever codepath sets that flag also runs this check. On mismatch, log at `ERROR`, refuse to flip `assume_valid_verified = true`, and surface the discrepancy on stderr. Do NOT panic: a running node that has already validated blocks on a correct chain should not self-destruct on detecting a build-time constant error, but it MUST refuse to use the constant for any subsequent path 2b bootstrap comparison. Subsequent tip-validations that would have taken path 2b fall through to `--verify-all`-equivalent behaviour (anchor at genesis, slow but correct).
   - **Build-time test.** Under a `#[test]` in `src/types/mod.rs` or a dedicated `tests/assume_valid_consistency.rs`, load a canonical chain fixture through `ASSUME_VALID_HEIGHT` and assert `storage.get_cumulative_work_at_height(ASSUME_VALID_HEIGHT) == ASSUME_VALID_CUMULATIVE_WORK` and `storage.get_block_id_at_height(ASSUME_VALID_HEIGHT) == ASSUME_VALID_HASH`. Prevents regenerated-but-not-updated constants from landing.
   - **Release-cycle procedure.** When the checkpoint constants are bumped (future release, new `ASSUME_VALID_HEIGHT`), all three values (`HEIGHT`, `HASH`, `CUMULATIVE_WORK`) must be regenerated together from the same canonical-node snapshot, and the build-time test re-run against a fresh fixture. Add this to the release checklist for any release that changes the checkpoint.

5. Only after step 4 completes, mark `confirmed: true`, set `peer_tip.cumulative_work = verified_cumulative_work`, and allow IBD from this peer.

**Why this closes the attack decisively.** The earlier draft's weakness was that for headers above our tip we could only bound difficulty (via `K`) rather than determine it exactly. With a common ancestor, the retarget function is a pure function of the anchor's state + height, so we can compute the exact required target at every forward height. An attacker producing an easy-PoW header is now rejected at difficulty check, not merely rate-limited by a cap. The cumulative-work factor (`0.5`) becomes unnecessary because the forward chain's PoW is consensus-exact: summed work is the genuine required work to have produced the chain.

### Pre-validation carry into IBD (avoids double-Argon2 cost)

Forward-header validation already runs full Argon2id on every `h_i` in the range. If we then drop that validated-header state and the IBD phase re-runs Argon2id when those same headers arrive inside blocks, we pay the Argon2 cost twice — unacceptable for long `Δheight` validations.

Fix: reuse the existing `pre_validated` flag on `PeerEvent` / `BlockMsg`. When a header is validated during tip-confirmation, cache the `(block_id, height, difficulty_target, pow_verified=true)` tuple in a per-peer (or global, expert's call) validated-headers map keyed by block_id. During IBD:

- When a block arrives whose header matches a cached validated-header entry (exact `block_id` match), mark its `PeerEvent` with `pre_validated = true`. The block-validation path already honours `pre_validated` to skip Argon2 re-evaluation (still runs body / merkle / tx validation — pre_validated only skips the PoW eval).
- Cache entry lifetime: valid for this peer's IBD session; evicted when IBD completes, fails, or the peer disconnects. Cache size is bounded by the attempt's wall-clock deadline and the single-attempt-per-peer + 4-global concurrency caps (see "Header-fetch bounds" below), not by a fixed entry count. Memory per entry ≈ 100 bytes. Worst-case under assume_valid cold bootstrap ≈ `chain_height − ASSUME_VALID_HEIGHT` entries ≈ 165K × 100 bytes = ~16 MiB per peer × 4 concurrent = ~64 MiB global peak. Acceptable on server-class nodes; monitor during implementation.
- Consensus safety: the cache is advisory-only for skipping Argon2 recomputation. If a block's header fails ANY other validation (merkle root, tx validation, prev linkage mismatch), it still gets rejected — the cache never admits an invalid block. The only bypassed work is Argon2 of a header whose PoW was already verified under exactly the same difficulty constant.

**Reviewer: expert should confirm the cache scope (per-peer preferred — easier lifetime management) and whether any existing code paths make assumptions about Argon2 always being run in-block (we know of none, but this is the consensus-adjacent risk worth a second pass).**

### Snapshot-and-recheck local anchor before flipping `confirmed: true` (new, per review)

Between `t_0` (start header-chain download from peer) and `t_1` (flip `confirmed: true` after all headers validate), our local chain may advance via blocks arriving from other peers. If a reorg during the window displaces the anchor from our canonical chain, the validated forward chain would no longer bridge into our actual tip, and flipping `confirmed: true` with the stale anchor could mis-direct IBD.

Guard sequence at `t_1`, serialized in the sync manager's single event-loop task (no named "sync-state lock" exists — serialization is achieved by the fact that tip-confirmation state transitions happen inside one `tokio` task processing `PeerEvent`s in order; the anchor re-check reads `self.tip` and peer tip state from within that task so no concurrent transition can interleave):

1. **Re-read local chain.** Get current canonical `block_id` at `anchor.height`.
2. **Anchor-stability check.** If the canonical block at `anchor.height` is still `anchor.block_id`, proceed: flip `confirmed: true`, set `active_ibd_peer`, release lock.
3. **Anchor moved (reorg during validation).** Two sub-cases:
   - **Peer's chain still descends from current canonical.** The reorg happened below the anchor, and the anchor is now on a sibling branch that's strictly worse. If `peer_tip.cumulative_work > our_tip.cumulative_work`, restart validation from a new common ancestor (re-run step 2 of "Required fix" above). Do not treat this as a peer failure.
   - **Peer's chain is now orphaned relative to ours.** The reorg went deeper than anchor and our chain is now on a different branch. Abort this tip-confirmation; do NOT flip `confirmed: true`. Peer can re-assert tip on next cycle.
4. **Do NOT record strikes for anchor-moved-due-to-local-reorg.** That's our movement, not peer misbehaviour.

This closes a real but narrow race. In steady state the anchor is essentially stable (our tip advances once per ~10s, validation completes in seconds), so the re-validation case fires rarely. Integration test coverage below.

### Checkpoint interaction (resolved per review)

The node has `ASSUME_VALID_HEIGHT = 130_000` with hardcoded `ASSUME_VALID_HASH`. Resolution:

- **Default build (assume_valid enabled):** If our local tip is at or past the checkpoint, anchor at the common ancestor between peer's chain and our fully-validated chain (checkpoint irrelevant in this case — we already have real state). If our local tip is BELOW the checkpoint (cold bootstrap or early IBD), permit the common ancestor to be the checkpoint block itself — trust the checkpoint hash as a known-valid anchor because we already trust it elsewhere in the codebase. This is **Option Y** from the earlier draft.
- **`--verify-all` / `--no-assume-valid` build:** Anchor strictly at the common ancestor with our fully-validated chain (genesis or higher). No checkpoint shortcut. Matches user intent when they pass the flag.
- **Checkpoint validity:** regardless of mode, the checkpoint only helps as an anchor below our tip. For claimed-tip validation *above* our tip, exact forward-difficulty checking runs from the anchor (whichever it is) up to the peer's claim — no checkpoint shortcut above our local tip, ever, because the checkpoint does not authorize any specific future chain.

**Why Y + fallback:** the checkpoint is a trust anchor we already rely on for IBD speed; re-deriving it from scratch in the sync-layer adds cold-bootstrap round-trips without new security. But a user who opted out of assume_valid at build time explicitly asked to re-validate from genesis — honour that.

### Batch size for header requests (resolved per review)

The existing protocol constant is `MAX_GETBLOCKS_ITEMS = 64`, which applies to both `GetBlocks` and `GetHeaders`. A full validation of a 100K-block height delta at batch 64 = ~1,562 round-trips, which is expensive on cold bootstrap but OK thereafter (and bounded).

Resolution: **keep `MAX_GETBLOCKS_ITEMS = 64` for this release.** Do not raise as part of v1.5.0; a protocol-constant bump would be a deliberate separate change (potential wire-compat concern, worth its own review cycle). At 64-per-batch + ~100ms round-trip, worst-case cold bootstrap fetch of a 165K above-checkpoint span = ~2,578 × 100ms = ~4.3 minutes of header-fetch time. Header fetch is **not** the bottleneck — Argon2 validation is (see cold-bootstrap cost analysis under "Header-fetch bounds" below).

**Follow-up (out of scope for v1.5.0):** propose a protocol bump to `MAX_GETBLOCKS_ITEMS = 512` in a subsequent release, version-gated. Not blocking for v1.5.0.

### Header-fetch bounds per tip-validation attempt (revised per review — no hard height cap)

The previous draft proposed `MAX_VALIDATION_HEADERS = 100_000` as a hard height ceiling. Expert correctly flagged this breaks honest cold bootstrap at `tip > ASSUME_VALID_HEIGHT + 100_000 = 230_000` (current tip is already ~295K). Removed.

Replacement: **bound the work, not the height.** Bounds are applied per-attempt and globally:

- **Per-attempt wall-clock timeout (revised, CPU-scaled):** the deadline is computed at attempt start as `max(7200, ceil(expected_argon2_seconds × 1.5))`, where `expected_argon2_seconds = expected_header_count / effective_rate_per_sec`. `effective_rate_per_sec` is the regime's current rate cap (see cold-bootstrap / steady-state regimes below). `expected_header_count` is `peer_claimed_height - anchor_height` (known once the anchor is located). This means a 1-core cold-bootstrap node validating 165K headers at `MAX_VALIDATION_ARGON2_PER_SEC_BOOTSTRAP = 10` gets a ~6.9h deadline (1.5× 16,500s), not 2h. A 4-core machine gets ~1.7h. The fixed 2h floor catches short attempts that still deserve a reasonable ceiling. The explicit formula replaces the previous fixed `TIP_VALIDATION_DEADLINE_SECS = 7200` constant.
- **Per-attempt header cap:** no fixed limit. The attempt terminates naturally when the peer's claimed tip height is reached, OR the wall-clock timeout fires, OR the peer fails a round-trip.
- **Per-peer concurrency:** 1 concurrent tip-validation per peer. A second tip claim from the same peer mid-validation aborts the earlier attempt and starts fresh — prevents an attacker cycling claims to multiply our workload.
- **Global concurrency (revised, regime-dependent):** in the **cold-bootstrap regime** (see below), `MAX_CONCURRENT_TIP_VALIDATIONS_BOOTSTRAP = 1`. Only one tip-validation at a time is permitted; additional claims queue and are processed after the active one finishes or aborts. This makes the "single active validation" rationale for the bootstrap rate-cap regime an explicit invariant rather than a hopeful claim. In the **steady-state regime**, `MAX_CONCURRENT_TIP_VALIDATIONS = 4` as before.
- **Round-trip timeout:** introduce `TIP_VALIDATION_BATCH_TIMEOUT_SECS = 120` (matches the inline `Duration::from_secs(120)` used by the existing IBD path at `sync.rs:4117`; named as a new constant for this feature rather than extracting the IBD inline). Batch-timeout aborts the attempt but does NOT record a strike (strikes only on delivered-but-invalid, see alt C below).

**Cold-bootstrap cost analysis (corrected per review).** Under assume_valid (default), the checkpoint at height 130,000 is treated as the anchor. Above-checkpoint headers at current chain height ~295K = ~165K headers to validate. At the proposed rate cap of `MAX_VALIDATION_ARGON2_PER_SEC = 20`, Argon2 time dominates: 165K / 20 ≈ **8,250 seconds ≈ 2.3 hours**. Header fetch at batch 64 with 100ms RTT = ~4.3 min, negligible by comparison. This is the real cost, not the ~200s I previously wrote.

**Resolution (expert's call): Option P.** The validation layer has two regimes, selected at each attempt's start from the node's local tip height. Each regime specifies BOTH the Argon2 rate cap AND the concurrency cap (they are a coupled pair, not independent knobs):

- **Cold-bootstrap regime (our local tip ≤ `ASSUME_VALID_HEIGHT` under assume_valid, or our local tip is genesis under `--verify-all`):**
  - `MAX_VALIDATION_ARGON2_PER_SEC_BOOTSTRAP = num_cpu::get().max(1) × 10` evals/sec. Per-core rate chosen so the Argon2 cost of Argon2id(m=64MiB, t=2) at ~100ms/eval saturates one core at 10 evals/sec.
  - `MAX_CONCURRENT_TIP_VALIDATIONS_BOOTSTRAP = 1`. This is what makes the rate-cap choice safe: there is by construction one validation in flight, additional tip claims queue. Without this concurrency=1 invariant, four 40-eval/sec attempts could consume 160 evals/sec, contradicting the "single active validation" rationale.
  - Per-peer still 1 concurrent.
  - Wall-clock deadline scales with expected work (see "Per-attempt wall-clock timeout" above): ~6.9h on a 1-core node for 165K headers, ~1.7h on 4 cores, ~26 min on 16 cores.

- **Steady-state regime (our local tip > `ASSUME_VALID_HEIGHT`):**
  - `MAX_VALIDATION_ARGON2_PER_SEC = 20` evals/sec applies globally across all concurrent tip-validation attempts. Keeps the adversarial flood ceiling low — at most ~2 cores of CPU burned on validation under worst-case peer churn.
  - `MAX_CONCURRENT_TIP_VALIDATIONS = 4`.
  - Per-peer still 1 concurrent.

The transition is automatic: each validation attempt reads the node's local tip height at start and uses the corresponding regime (both rate cap and concurrency cap). An attempt that straddles the checkpoint (node's tip passes `ASSUME_VALID_HEIGHT` mid-bootstrap) stays in the regime it started in for that attempt's duration; subsequent attempts re-check and flip regime.

### Alternative / complementary mitigations (revised)

Per expert direction, (B) and (D) are dropped — not appropriate or no longer needed.

**(A) — rewritten as validation-work budget.** Replace the earlier height-distance cap with a **total Argon2id validation budget** across all concurrent tip-validations: `MAX_VALIDATION_ARGON2_PER_SEC = 20`. This is a rate limit on the validation-layer's CPU cost, independent of how many peers are trying to get us into IBD. At 20 evals/sec and ~100ms per eval, that's 2 validation-CPU-seconds per wall-clock second, i.e., ~2 cores of headroom on Argon2. Chosen to be much less than legitimate miner hashrate (~6,250 evals/s network-wide, ~100-1000/s for an active miner node) so a flood of validation requests cannot starve normal mining or block validation. Excess validation requests queue up to a 60-second ceiling before being rejected outright.

**(C) — narrowed: strikes on delivered-but-invalid headers only, using the existing strike mechanism.** When a peer delivers a header that fails forward-chain validation (wrong difficulty, wrong PoW, broken chain, or checkpoint-hash mismatch in the cold-bootstrap subpath), call the existing `record_ip_strike(ip, Some(identity))` (`src/network/sync.rs:795`) with the authenticated post-handshake `identity`. The brief does NOT introduce a separate "identity strike table" or `N_STRIKES` constant — it reuses the existing IP-strike accumulator, which already bans the identity as a side effect when the IP crosses the existing strike threshold (see `sync.rs:924`). This is deliberate: (a) the mechanism already works and is exercised by other sync-layer error paths (e.g., `sync.rs:2776, 2975, 3133, 3257, 3389, 3947, 3957, 3996, 4055, 5032`); (b) adding a parallel identity-only strike table would duplicate state and produce inconsistent eviction decisions across path-dependent error paths. Do NOT strike on generic IBD failure that might be legit flake (peer disconnect, timeout delivering a later batch).

**(B) — DROPPED.** Parallel IBD probes is a separate concern and out of scope; v1.5.0 focuses on making single-peer tip-confirmation safe. Parallel IBD can be revisited later if needed.

**(D) — DROPPED.** With exact-forward validation requiring real consensus work from common ancestor up, the "tiny epsilon IBD trigger" vector is structurally eliminated — an attacker cannot present a validated chain with negligible cumulative work because the work is summed from actual per-block required difficulty. The earlier `X = 0.01` threshold is no longer needed.

### Scope and non-goals

- **Consensus path is NOT touched.** Block validation in `process_block_event` is unchanged. This change affects only the sync decision: when do we start IBD, from whom.
- **No changes to PoW verification itself.** Argon2id evaluation remains identical.
- **No changes to difficulty retargeting *logic*.** The retarget math in `consensus/difficulty.rs:56-98` (parent target, retarget-window lookback, `retarget` clamp) is unchanged. The change adds a new `expected_difficulty_overlay` entry point that runs the same algorithm over a `ForwardHeaderOverlay` instead of `ChainStorage` — or, equivalently, refactors `expected_difficulty` to take a `HeaderSource` trait with two impls. Either factoring preserves the algorithm byte-for-byte.
- **No changes to block/tx validation.** This is purely sync-layer hardening.
- **Not a ban mechanism.** Peers that fail the new validation are rejected for tip-confirmation purposes but can stay connected for other traffic (blocks, tx gossip, peer exchange). Existing strike/ban logic applies separately.

### Error semantics

**Unified strike policy (narrowed per review):** a strike is recorded ONLY when a peer delivers a header that fails forward-chain validation — wrong difficulty target, invalid PoW, broken chain integrity, a header whose block_id contradicts its position, or (in cold-bootstrap subpath 2b) a checkpoint-height header whose block_id does not equal `ASSUME_VALID_HASH`. Timeouts, disconnects, and "gave up delivering later batches" are NOT strikes: those can be honest peer flake, and over-striking would let an attacker frame honest peers by spoofing their traffic. This matches alt C below. The strike is recorded via the existing `record_ip_strike(ip, Some(identity))` — no new strike table is added.

- **Delivered-but-invalid header** (wrong target, bad PoW, broken `prev_block_id`, wrong height, or wrong checkpoint hash): do NOT mark `confirmed: true`. Log at `INFO`: `"Peer {id} claimed tip at height H failed chain-validation: {specific reason}"`. **Call `record_ip_strike(ip, Some(identity))`.** Accumulation may ban the IP (and the associated identity as a side-effect) via the existing mechanism.
- **Header chain download times out mid-batch:** abort this validation attempt; tip stays unconfirmed. **No strike.** The peer can try again on the next tip announcement.
- **Peer disconnects mid-validation:** give up on this attempt; tip stays unconfirmed. **No strike.** Peer goes back into the normal cycle.
- **Anchor moved due to local reorg during validation** (see snapshot-and-recheck section): restart with new anchor or abort, peer's fault not attributed. **No strike.**
- **No new error codes on the wire protocol.** All the new validation is internal to our node.

### Rollout considerations

- **Wire-protocol compatible.** The new validation uses existing `GetHeaders`/`HeadersResponse` messages with `MAX_GETBLOCKS_ITEMS = 64` unchanged. No protocol version bump needed.
- **Asymmetric legit behaviour.** A legitimate peer responding to our forward-chain `GetHeaders` requests delivers more headers per new-peer handshake than today. Worst case is honest cold bootstrap above the checkpoint: ~165K headers × 156 bytes = ~25.7 MiB per attempt. Typical steady-state validation (`Δheight ≈ 10–1000`) is ~1.5 KiB to ~156 KiB — negligible relative to block traffic.
- **Worst case under adversarial load.** Bounded by the validation-work budget (`MAX_VALIDATION_ARGON2_PER_SEC = 20`, 4 concurrent slots). An attacker flooding fake tips cannot exceed this — excess queues up to 60s before being rejected. CPU cost: ~2 cores sustained worst case, well below normal mining/validation headroom.
- **Cold bootstrap cost (scales with CPU).** With assume_valid enabled the anchor is the checkpoint at height 130K (fetched and hash-verified against `ASSUME_VALID_HASH` via cold-bootstrap subpath 2b), so only above-checkpoint headers (~165K at current chain tip) need Argon2 validation. Header fetch at batch 64 + 100ms RTT ≈ ~4.3 min (not the bottleneck). Argon2 cost dominates. Per the selected rate-cap regime (Option P), during cold bootstrap the cap is `num_cpu::get().max(1) × 10` evals/sec — concurrency is also forced to 1 so the rate cap is the real ceiling. Approximate wall-clock cost: **1 core ≈ 4.6h, 2 cores ≈ 2.3h, 4 cores ≈ 70 min, 8 cores ≈ 35 min, 16 cores ≈ 17 min.** The per-attempt deadline scales with `1.5 × expected_argon2_seconds`, so even a 1-core node gets a ~6.9h deadline (not the previous hard 2h). Once the node's tip passes the checkpoint, steady-state cap (`MAX_VALIDATION_ARGON2_PER_SEC = 20`, concurrency 4) applies. **Implication: single-core VPS machines will take several hours to cold-bootstrap. This is acceptable (one-time cost, runs in background) but should be called out in release notes.**
- **Chain reorg interaction.** Addressed by the snapshot-and-recheck-anchor step before flipping `confirmed: true` (see above). Normal post-IBD reorg handling on already-downloaded blocks is unaffected. Reviewer: please confirm separability.
- **Expert re-review on the serialization section** where `confirmed: false → true` transitions happen — specifically the snapshot-recheck step. Assumed property: sync manager processes `PeerEvent`s in a single task, so tip-state mutations do not interleave. Reviewer should confirm this assumption holds for the paths touched (`attach_session` → `TipResponse` handler → forward-chain validation → snapshot-recheck → flip confirmed).
- **pre_validated cache lifecycle.** Per-peer, evicted on peer disconnect or IBD completion. No persistent storage — cache is in-memory only, so node restart always re-validates. Intentional: keeps the cache strictly advisory and bounded in lifetime.
